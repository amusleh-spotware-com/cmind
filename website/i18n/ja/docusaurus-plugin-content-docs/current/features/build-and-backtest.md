# Build & backtest cBots

cTrader cBotを構築、実行、バックテストします (C# **および** Python、両方とも.NET) ブラウザ内のMonaco IDEから、公式 `ghcr.io/spotware/ctrader-console` イメージで実行。

## Build

- **Builder** ページはMonaco エディタをホストします。`CBotBuilder` は `dotnet build` で**使い捨てコンテナ内**で (`AppOptions.BuildImage`、作業ディレクトリは `/work` にバインドマウント)、プロジェクトをコンパイルするため、信頼されないユーザーのMSBuildがホストに到達しません。NuGet復元は共有ボリュームを介してビルド間で キャッシュされます。Webホストはdocker ソケットアクセスが必要です。
- C# と Python スターターテンプレートは `src/Nodes/Builder/Templates/` に格納されています。

## Run & backtest

- **Instances** = TPH 状態階層 (`Run`/`Backtest` × `Pending`/`Scheduled`/`Starting`/
  `Running`/`Stopping`/`Stopped`/`Failed`)。遷移はエンティティを置き換え (IDが変更)、
  コンテナ ID は引き継がれます。
- `NodeScheduler` は最も負荷の少ない適切なノードを選択します。`ContainerDispatcherFactory` は
  リモートノード HTTP エージェントまたはローカル Docker ディスパッチャーにルーティングします。
- 完了ポーラーは終了したコンテナを調整します (バックテストコンテナは `--exit-on-stop` を介して自動終了)。レポートが存在 → 完了 (store `ReportJson`)、 存在しない → 失敗。
- ライブコンテナログはSignalRを介してブラウザにストリーミングされます。バックテストのエクイティカーブはレポートから解析されてチャート化されます。

## Backtest market data is cached per account

cTrader Console は履歴ティック/バーデータを `--data-dir` にダウンロードします。そのディレクトリは**安定した永続的なキャッシュ (トレーディングアカウント (その口座番号) をキーとする)** — ノードのディスクからそのコンテナパスにバインドマウント (`/mnt/data`)、**インスタンス単位の作業ディレクトリから分離された非ネストマウント**。そのため、同じアカウントの各バックテストは、各実行ごとに再ダウンロードするのではなく、既にダウンロードされたデータを**再利用**します。(以前は、データディレクトリは インスタンス単位の作業ディレクトリの下に格納されていたため、IDは各実行ごとに変更されていたため、各バックテストで新しいダウンロードが強制されていました。) 一時的なインスタンス単位の作業ディレクトリには、アルゴ、パラメーター、パスワード、レポートが保持されます。共有データキャッシュはノードのバックテストデータ使用量にカウントされ、ノードクリーンアクションでクリアされます。

## Backtest settings

**Backtest** ダイアログは、cTrader Console バックテスト CLI が受け入れるすべての設定を公開するため、コマンドラインに触れることはありません。

- **From / To** — バックテストウィンドウ (`--start` / `--end`)。
- **Data mode** — 3つのcTraderモードのいずれか (`--data-mode`): **Tick data** (`tick`、正確)、
  **m1 bars** (`m1`、高速)、または **Open prices only** (`open`、最速)。
- **Starting balance** — デフォルトは `10000` (`--balance`)。**0の残高は取引を行わず、cTraderが空のレポートを出力してからクラッシュします** ("Message expected")、そのため常にゼロ以外の残高が送信されます。
- **Commission** および **Spread** — `--commission` / `--spread` (スプレッドはピップス単位)。
- **Data file** (オプション) — 履歴データファイルへのノード側パス (`--data-file`)。ダウンロード/キャッシュされたデータを使用する場合は空のままにします。
- **Expose environment variables** — ホスト環境変数をcBotに渡すトグル (the `--environment-variables` flag)。

## Instance detail page

インスタンスを開く (`/instance/{id}`) と、そのライブステータス、ログ、およびバックテストの場合はエクイティカーブが表示されます。**ブラウザタブタイトル**は特定のインスタンスを反映します (**cBot名 · 種類 · シンボル**、例えば `TrendBot · Backtest · EURUSD`)、そのためライブ実行タブとバックテストタブは一目で区別できます。同じcBotの実行とバックテストは別の**系統** (状態遷移全体で実行される安定した系統ID) として追跡されるため、ページは正確に1つのインスタンスに従い、実行のデータとバックテストのデータを混在させることはありません。

## Instance lifecycle controls

各インスタンス行 (およびその詳細ページ) には状態正確なコントロールがあります。**アクティブな**インスタンスは **Stop** を表示します。**ターミナル** (Stopped / Completed / Failed) のインスタンスは **Start (▶)** を表示し、同じcBot、アカウント、シンボル、タイムフレーム、パラメーターセット、イメージで再起動します (実行は実行として再起動し、バックテストはバックテストとして再起動します)。Stopをクリックすると、「Stopping…」という通知が表示され、それが解決するまでアイコンが無効になり、新しく作成された実行がリストに直ちに表示されます — ページを再ロードする必要はありません。

コンソールログは、インスタンスが終了したときに**永続化**されます — 実行 (停止時) とバックテスト (完了時) の両方で — そのため、最後の実行のログは詳細ページで表示可能なままであり、ログツールバーを介して、**クリップボードにコピー** (Copyアイコン) または **ダウンロード** (Downloadアイコン) でき、コンテナが削除されても表示できます。どちらもインスタンスのフルコンソールログで機能し、画面上のテールだけではありません。

アップロードされた `.algo` はここで構築されたことがないため、その**Last Build**列 (cBots ページ) は空白のままです (ブラウザで構築したcBotのみの構築時間を表示します)。

## Edit & re-run a stopped instance

**停止**インスタンス (実行またはバックテスト) には**Edit** コントロールがあります — リストの行にあるアイコン**および**その詳細ページのStart/Stopの隣 — 現在の構成で**事前入力**されたダイアログを開きます。**トレーディングアカウント、シンボル、タイムフレーム、パラメーターセット、イメージタグ**を変更できます (バックテストの場合、**ウィンドウと上記のすべてのバックテスト設定**)、**Save & start** で新しい設定で再起動します (停止したインスタンスを置き換えます)。コントロールは、**インスタンスがアクティブな間は無効**です — 停止したインスタンスのみが編集できます。

## Run from the code editor

コードエディターで **Run** をクリックすると、ブラインド、ハードコード実行を行わずにダイアログが開きます。

- **Trading account** (必須) — cBotが接続するcTraderアカウント。
- **Parameter set** (オプション) — 既存のセットを選択するか、空のままにしてcBotの**デフォルトパラメーター値**で実行します。セレクターの横にある**+** ボタンは新しいパラメーターセットをインラインで作成します (下記を参照) とそれを選択します。
- **Symbol / Timeframe** はデフォルトで `EURUSD` / `h1` で、変更できます。**Cancel** または **Run**。

**Run** でエディターは現在のソースを保存+構築し、選択したアカウント上でインスタンスを開始し、選択したパラメーターで、ライブコンテナログをテールします。(ログストリームは署名ユーザーの認証cookie を `/hubs/logs` SignalR ハブに転送するため、接続して `Invalid negotiation response received` で失敗するのではなく。)

## Parameter sets

**Parameter set** は、cBotパラメーターオーバーライドの名前付けされた再利用可能なセットで、フラットJSONオブジェクトとして保存されます。各パラメーター名をスカラー値にマッピングします。例えば `{"Period": 14, "Label": "trend"}`。実行/バックテスト時には、cTrader `params.cbotset` ファイルに変換されます (`{ "Parameters": { … } }`)。cBotの**Parameter sets** ダイアログから生のJSONとして、またはRunダイアログからインラインとしてセットを作成/編集できます。

すべてのパラメーターセット**はcBotに属しており**: New Parameter Set ダイアログには、すべてのcBotが一覧表示され、**1つを選択する必要があります** — cBotが選択されるまで作成がブロックされます。セットの**名前はcBot単位でユニークです**: セットを作成または名前変更して、同じcBotの別のセットが既に使用している名前は拒否されます (ダイアログの明確なエラー、APIで `409 Conflict`)。同じ名前は**別の** cBotで再利用できます。

JSONは**保存時に検証**されます。単一のフラットオブジェクトであり、その値はすべてスカラーである必要があります (string / number / bool)。非オブジェクトルート、配列、ネストされたオブジェクト、`null` 値、または不正なJSONは拒否されます (ダイアログの明確なエラー、APIで `400 Bad Request`)。空のオブジェクト `{}` は許可されており、「オーバーライドなし」を意味します。

## cTrader Console CLI notes

バックテストは `--data-mode` が必要です (デフォルト `m1`)、日付は `dd/MM/yyyy HH:mm` として、
`params.cbotset` JSON 位置引数。 `run` は `--data-dir` を拒否します (バックテストのみ)。
`ContainerCommandHelpers` を参照してください。

## Nodes & scale

実行容量はノードエージェント (自己登録+ハートビート) を追加することで スケーリングされます。
[node discovery](../operations/node-discovery.md) および [scaling](../deployment/scaling.md) を参照してください。

## A trading account is required

cBotの実行またはバックテストには、cBotが接続するcTraderトレーディングアカウントが必要です。**Trading accounts** の下に追加するまで、**Run New cBot** / **Backtest New cBot** ボタンは無効です (ツールチップ付き) で、ページはアカウント設定へのリンクを表示するプロンプトを表示します — もはや生の `stream connect failed` エラーには達しません。アカウントのないボットから。
