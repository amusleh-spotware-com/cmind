# Backtest Integrity Lab

Οι λιανικές πλατφόρμες σας δείχνουν το Sharpe ή το καθαρό κέρδος ενός backtest και σταματούν εκεί. Τα
ιδρύματα ποτέ δεν εμπιστεύονται ένα raw backtest — ρωτούν αν το αποτέλεσμα επιβιώνει **διόρθωση για
selection bias και τον αριθμό διαμορφώσεων που δοκιμάστηκαν**. Το Backtest Integrity Lab φέρνει αυτόν
τον έλεγχο στο cMind. Είναι **ντετερμινιστικά μαθηματικά** (χωρίς AI, χωρίς εξωτερικές κλήσεις), οπότε η
ετυμηγορία είναι αναπαραγώγιμη και κάθε αριθμός εξηγήσιμος.

Ανοίξτε το στο **cBots → Integrity** (`/quant/integrity`).

## Τι υπολογίζει

Δεδομένης μιας σειράς αποδόσεων (ή μιας καμπύλης equity/υπολοίπου) και του αριθμού συνόλων παραμέτρων
που δοκιμάσατε για να φτάσετε σε αυτό, ο αναλυτής αναφέρει:

- **Sharpe ratio** — ανά περίοδο και ετήσιο (τετραγωνική ρίζα του χρόνου).
- **Probabilistic Sharpe Ratio (PSR)** — η εμπιστοσύνη ότι το *αληθινό* Sharpe υπερβαίνει το
  benchmark, λαμβάνοντας υπόψη μήκος ιστορικού, ασυμμετρία και κύρτωση (Bailey & López de Prado,
  2012). Ένα μικρό ή με χοντρές ουρές ιστορικό το μειώνει.
- **Deflated Sharpe Ratio (DSR)** — PSR μετρημένο έναντι ενός **εκλυμένου benchmark**: το Sharpe
  που θα περιμένατε από το *καλύτερο από N τυχαίες δοκιμές* υπό την μηδενική υπόθεση (the False
  Strategy Theorem). Όσες περισσότερες διαμορφώσεις δοκιμάσατε, τόσο υψηλότερο το όριο — αυτό
  πιάνει το overfitting.
- **t-statistic** της μέσης απόδοσης. Ακολουθώντας τους Harvey, Liu & Zhu, μια γνήσια πλεονεκτική
  στρατηγική πρέπει να ξεπερνά **t ≥ 3.0**, όχι το βιβλιακό 2.0.
- **Ασυμμετρία / κύρτωση** των αποδόσεων, που τροφοδοτούν τις διορθώσεις PSR/DSR.

## Η ετυμηγορία

| Ετυμηγορία | Σημασία | Κανόνας |
|---|---|---|
| **Robust** | Η πλεονεκτική συνέχειες ισχύει για τις δοκιμές που εκτελέσατε. | DSR ≥ 95% **και** PSR ≥ 95% **και** \|t\| ≥ 3.0 |
| **Fragile** | Στατιστικά ενεργή αλλά όχι πειστικά — μην αυξήσετε τις θέσεις μόνο με αυτό. | μεταξύ των δύο |
| **Overfit** | Πιθανότατα τεχνούργημα του selection bias, όχι πραγματική πλεονεκτική. | DSR < 90% |

Κάθε αποτέλεσμα φέρει απλή επεξήγηση για να μην κρύβεται ποτέ το "γιατί".

## Probability of Backtest Overfitting (κατά δοκιμές)

Η τροφοδοσία ενός αριθμού δοκιμών είναι καλή· η τροφοδοσία της **πραγματικής out-of-sample σειράς
κάθε διαμόρφωσης που δοκιμάσατε** είναι καλύτερη. Επικολλήστε τις στην προαιρετική **trial grid**
(μια σειρά ανά γραμμή) και το cMind εκτελεί **Combinatorially-Symmetric Cross-Validation** (Bailey,
Borwein, López de Prado & Zhu, 2015): χωρίζει τις παρατηρήσεις σε ομάδες, και για κάθε τρόπο
επιλογής του μισού ως in-sample επιλέγει την καλύτερη in-sample διαμόρφωση και ελέγχει αν αυτός ο
νικητής βρίσκεται στο κάτω μισό **out-of-sample**. Η **Probability of Backtest Overfitting (PBO)**
είναι το κλάσμα των διασπάσεων όπου ο νικητής απέτυχε να γενικευτεί. Μια PBO κοντά στο 0 σημαίνει
ότι η καλύτερη διαμόρφωση είναι πραγματικά η καλύτερη· μια PBO 0.5 ή περισσότερο σημαίνει ότι η
διαδικασία επιλογής σας επιλέγει θόρυβο — η ετυμηγορία γίνεται **Overfit** ανεξάρτητα από το πόσο
καλός έμοιαζε ο νικητής.

```http
POST /api/quant/pbo
{ "trials": [[...], [...], ...] }
```

Όταν ο φυσικός βελτιστοποιητής cTrader Console προσγειωθεί, το cMind θα τροφοδοτεί αυτόματα την
πλήρη επιφάνεια δοκιμών εδώ.

## Trials — ο αριθμός που έχει σημασία

`Trials` είναι **πόσα σύνολα παραμέτρων δοκιμάσατε** πριν επιλέξετε αυτό. Η δοκιμή μιας στρατηγικής
και η δοκιμή δέκα χιλιάδων και η διατήρηση της καλύτερης είναι δύο τελείως διαφορετικά πράγματα: το
δεύτερο κατασκευάζει ένα υψηλό in-sample Sharpe τυχαία. Η τροφοδοσία της ειλικρινούς δοκιμαστικής
μέτρησης είναι όλο το νόημα — αυξάνει τον εκλυμό και μπορεί να μετακινήσει ένα "τέλειο" backtest
σε **Overfit**. Όταν ο φυσικός βελτιστοποιητής cTrader Console προσγειωθεί, το cMind τροφοδοτεί
αυτόματα το πραγματικό μέγεθος grid του sweep.

## Εισροές

- **Περιοδικές αποδόσεις** — ένας αριθμός ανά περίοδο (π.χ. `0.01` = +1%). Τουλάχιστον δύο.
- **Καμπύλη equity / υπολοίπου** — το cMind παραγάγει τις διαδοχικές απλές αποδόσεις για εσάς.
- Ή εκτελέστε το απευθείας σε ένα ολοκληρωμένο backtest: `POST /api/quant/integrity/backtest/{instanceId}`
  διαβάζει την αποθηκευμένη καμπύλη equity της αναφοράς.

## API

```http
POST /api/quant/integrity
{ "returns": [0.006, 0.004, 0.006, ...], "trials": 250 }
```

Επιστρέφει την ετυμηγορία, όλες τις μετρικές και την αιτιολόγηση. `POST /api/quant/integrity/backtest/{id}`
εκτελεί την ίδια ανάλυση σε ένα ολοκληρωμένο backtest που ανήκει σε εσάς.

## Γιατί είναι αξιόπιστο

Οι στατιστικές είναι καθαρές συναρτήσεις στο domain core (`Core.Quant`) με μηδενικές εξαρτήσεις
υποδομής — δεν μπορούν να τεθούν εκτός από ένα δικτυακό σφάλμα, και είναι καρφωμένες από golden-vector
unit tests έναντι των δημοσιευμένων τύπων. Οι κλειστής μορφής κανονικές CDF/αντίστροφες
(Abramowitz-Stegun / Acklam), οπότε οι ίδιες εισροές πάντα δίνουν την ίδια ετυμηγορία.
