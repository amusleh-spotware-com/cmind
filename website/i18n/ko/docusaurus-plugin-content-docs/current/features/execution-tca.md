# 거래 비용 분석 (TCA)

실행 알파는 거래당 작고 수천에서 매우 큽니다 — 은행과 プロ펌 데스크가 에지를 유지하는 방식의 대부분입니다. TCA는 *결정*했을 때 가격과 실제로 달성한 가격 사이의 드리프트距離を 측정합니다.

**cBots → Execution Cost** (`/quant/tca`)를 엽니다.

## 무엇을 측정하는가

**도착(결정) 가격**, **사이드** 및 **체결** (가격 × 수량)을 주면 보고합니다:

- **평균 체결 가격 (VWAP)** — 실제로 얻은 볼륨 가중 가격.
- **슬리피지 (bps)** — 베이시스 포인트의 도착에서 VWAP로의 드리프트, **부호화되어 양수 숫자가 비용** (매수高于到着 또는 매도低于到着)이고 음수 숫자가 가격 개선입니다.
- **구현 부족** — 해당 비용을 가격 × 수량으로 표현: 주문에서 드리프트가 비용이 든 금액.

```http
POST /api/quant/tca
{ "arrivalPrice": 1.1000, "side": "Buy",
  "fills": [ { "price": 1.1010, "quantity": 100 }, { "price": 1.1020, "quantity": 100 } ] }
```

## 스마트 슬라이싱 (Almgren-Chriss)

비용 측정 외 Beyond, cMind는 그것을 *최소화*하도록 대형 주문을 계획할 수 있습니다. **cBots → Execution Schedule** (`/quant/execution`)은 **Almgren-Chriss 최적 실행 일정**을 빌드합니다: 총 수량, 슬라이스 수, 위험 회피, 변동성 및 임시 시장 영향을 주면 각 슬라이스에서 거래할 크기를 반환합니다. 더 높은 위험 회피는 일정을 **프론트로드**합니다 (타이밍 위험 절감); 위험 회피가 0이면 평평한 **TWAP**로 됩니다. 슬라이스의 합계는 항상 총계입니다.

```http
POST /api/quant/execution-schedule
{ "totalQuantity": 100, "slices": 5, "riskAversion": 2, "volatility": 0.02, "temporaryImpact": 0.1 }
```

## 왜 신뢰할 수 있는가

인프라 의존성 및 외부 호출이 없는 순수 결정론적 도메인 코드 (`Core.Execution`)입니다 — 매수/매도 비용 부호, 가격 개선, 제로 슬리피지, VWAP 집계 및 입력 가드에 대해 단위 테스트되었습니다. 이것은 실행 품질의 측정 절반입니다; 복사 엔진이 미러링된 주문의 비용을judge (그리고 스마트 슬라이싱으로 줄이는) 데 사용하는 동일한 부족 메트릭입니다.
