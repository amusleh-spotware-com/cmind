# cBotの構築とバックテスト

cTrader cBotを構築、実行、バックテスト（C# **および** Python、両方.NET対応）します。ブラウザ内Monaco IDEから実行し、公式 `ghcr.io/spotware/ctrader-console` イメージを使用します。

## 構築

- **Builder** ページはMonacoエディタをホストし、`CBotBuilder` は **一時コンテナで** `dotnet build` を使用してプロジェクトをコンパイルします（`AppOptions.BuildImage`、作業ディレクトリは `/work` にバインドマウント）。これにより信頼されていないユーザーのMSBuildはホストに到達しません。NuGet復元はビルド間で共有ボリュームを通じてキャッシュされます。WebホストはDockerソケットアクセスが必要です。
- C#およびPythonのスターターテンプレートは `src/Nodes/Builder/Templates/` に存在します。

## 実行とバックテスト

- **Instances** = TPH状態階層（`Run`/`Backtest` × `Pending`/`Scheduled`/`Starting`/`Running`/`Stopping`/`Stopped`/`Failed`）。遷移でエンティティを置き換え（IDが変更）、コンテナIDは引き継がれます。
- `NodeScheduler` は最も負荷の少ない適格ノードを選択します。`ContainerDispatcherFactory` はリモートノードHTTPエージェントまたはローカルDockerディスパッチャにルーティングします。
- 完了ポーラーは終了したコンテナを調整します（バックテストコンテナは `--exit-on-stop` で自己終了します）。レポートが存在→完了、存在しない→失敗としてマークします。
- ライブコンテナログはSignalRを通じてブラウザにストリーム配信されます。バックテストエクイティカーブはレポートから解析されてチャートされます。

## バックテスト市場データはアカウントごとにキャッシュされます

cTrader Consoleは履歴ティック/バーデータを `--data-dir` にダウンロードします。そのディレクトリは **トレーディングアカウント（そのアカウント番号）をキーとした安定した永続キャッシュ** です。ノードのディスクからそのコンテナパス（`/mnt/data`）にバインドマウントされており、インスタンスごとの作業ディレクトリから **分離した非ネストマウント** です。したがって同じアカウントでのすべてのバックテストは既にダウンロード済みのデータを **再利用** し、実行ごとに再ダウンロードしません。（以前はデータディレクトリがインスタンスごとの作業ディレクトリ下に存在し、実行ごとにIDが変更されたため、バックテストごとに新規ダウンロードが強制されていました。）エフェメラルなインスタンスごとの作業ディレクトリはアルゴ、パラメータ、パスワード、レポートを保持します。共有データキャッシュはノードのバックテストデータ使用量としてカウントされ、ノードクリーンアクションでクリアされます。

## バックテスト設定

**Backtest** ダイアログはユーザーが調整可能なcTrader Consoleバックテスト設定を公開するため、コマンドラインを触れる必要がありません。

- **Symbol / Timeframe** — タイムフレームは **cTraderのすべてのピリオドのドロップダウン**（`t1`…`t1000`、`m1`…`m45`、`h1`…`h12`、`D1`/`D2`/`D3`、`W1`、`Month1`、およびRenko/Range/Heikinピリオド）で、コンソールの正規形式で表示されるため、常に有効な `--period` を選択できます。
- **From / To** — バックテストウィンドウ（`--start` / `--end`）。
- **Data mode** — cTraderの3つのモード（`--data-mode`）の1つ：**Tick data**（`tick`、精密）、**m1 bars**（`m1`、高速）、または **Open prices only**（`open`、最高速）。
- **Starting balance** — デフォルトは `10000`（`--balance`）。**0残高ではトレードが行われず、cTraderは空のレポートを生成して「Message expected」でクラッシュします** ため、常に0以外の残高が送信されます。
- **Commission** — `--commission`。
- **Spread** — `--spread`、**0未満に設定できない数値フィールド（ピップス単位）**。**Tick dataモードでは非表示** です。cTraderはティックデータ自体からスプレッドを導出するため（`--spread` は送信されません）。

データディレクトリ（`--data-file` / `--data-dir`）はアプリ自体で管理されます（アカウントごとのキャッシュ（上記参照））。ダイアログでは公開されません。

:::note cTraderが空のバックテストでクラッシュする
バックテストが **何も結果を生成しない** 場合（トレードなし、または選択した日付/シンボルで市場データなし）、cTrader Consoleのレポートライター自体が `Message expected` をスローしてレポートなしで終了します。アプリはこの上流バグを修正できませんが、それを検出し、生のスタックトレースの代わりに、インスタンスを **Failed** にマークし、実用的な理由（「選択範囲に対するバックテスト結果なし…」）を表示します。利用可能な市場データを持つより広い日付範囲を選択して再試行してください。
:::

## インスタンス詳細ページ

インスタンスを開く（`/instance/{id}`）と、ライブステータス、ログ、およびバックテストの場合はエクイティカーブが表示されます。**ブラウザタブタイトル** は特定のインスタンス（**cBot名・種類・シンボル**、例：`TrendBot · Backtest · EURUSD`）を反映するため、ライブ実行タブとバックテストタブは一目で区別できます。同じcBotの実行とバックテストは別の **lineages** として追跡されます（状態遷移を通じて運ばれる安定したlineage ID）。ページは正確に1つのインスタンスのみを追跡し、実行のデータとバックテストのデータを混在させません。

## インスタンスライフサイクルコントロール

各インスタンス行（およびその詳細ページ）には状態に応じた正しいコントロールがあります。**アクティブな** インスタンスは **Stop** を表示します。**ターミナル** なもの（Stopped / Completed / Failed）は **Start（▶）** を表示し、同じcBot、アカウント、シンボル、タイムフレーム、パラメータセット、イメージで再起動します（実行は実行として再起動し、バックテストはバックテストとして再起動します）。Stopをクリックすると「Stopping…」の通知が表示され、解決するまでアイコンが無効になり、新しく作成された実行がリストに即座に表示されます。ページリロードは不要です。

コンソールログは **インスタンスが終了するとき永続化** されます。実行時（Stopで）、**バックテスト** 時（完了で）同様に、最後の実行のログは詳細ページで表示可能なままであり、ログツールバー経由で **クリップボードにコピー**（Copy logsアイコン）または **ダウンロード**（Download logsアイコン）でき、コンテナが消えた後も同様です。両方ともインスタンスのフルコンソールログに作用し、画面上のテールだけではありません。

**アップロード** された `.algo` はここで構築されたことがないため、cBotsページの **Last Build** 列は空白のままです（ブラウザで構築したcBotのみ構築時間を表示します）。

## 停止したインスタンスを編集して再実行

**停止したインスタンス**（実行またはバックテスト）は **Edit** コントロール（リスト内のそのあるいは詳細ページのStart/Stopの横にあるアイコン）があり、現在の設定で **事前入力** されたダイアログを開きます。**トレーディングアカウント、シンボル、タイムフレーム、パラメータセット、イメージタグ** を変更し（またはバックテストの場合、**ウィンドウとすべてのバックテスト設定** 上記）、**Save & start** で新しい設定で再起動します（停止したインスタンスを置き換えます）。このコントロールはインスタンスがアクティブな間は **無効** です。停止したインスタンスのみが編集可能です。

## コードエディタから実行

コードエディタで **Run** をクリックすると、ブラインド、ハードコードされた実行を実行する代わりにダイアログが開きます。

- **Trading account**（必須） — cBotが接続するcTraderアカウント。
- **Parameter set**（オプション） — 既存のセットを選択するか、cBotの **デフォルトパラメータ値** で実行するために空のままにします。セレクタの隣の **+** ボタンは新しいパラメータセットをインラインで作成し（下記参照）選択します。
- **Symbol / Timeframe** は `EURUSD` / `h1` にデフォルト設定され、変更できます。**Cancel** または **Run**。

**Run** 時、エディタは現在のソースを保存してビルドし、選択されたアカウントで選択されたパラメータを使用してインスタンスを開始し、ライブコンテナログをテールします。（ログストリームはサインインユーザーの認証クッキーを `/hubs/logs` SignalRハブに転送するため、`Invalid negotiation response received` で失敗する代わりに接続できます。）

## パラメータセット

**parameter set** は、フラットJSONオブジェクトとして保存された、名前付きの再利用可能なcBotパラメータオーバーライドのセットです。各パラメータ名をスカラー値にマッピングします。例：`{"Period": 14, "Label": "trend"}`。実行/バックテスト時、cTrader `params.cbotset` ファイルに変換されます（`{ "Parameters": { … } }`）。cBotの **Parameter sets** ダイアログからまたはRunダイアログからインラインでセットを作成/編集できます。

すべてのパラメータセットは **cBotに属します**：New Parameter SetダイアログはすべてのcBotをリストし、**1つを選択する必要があります**。cBotが選択されるまで作成はブロックされます。セットの **名前はcBotごとに一意です**：セットを別の名前に作成または名前変更するとき、同じcBotの別のセットが既に使用している場合は拒否されます（ダイアログ内で明確なエラー、APIで `409 Conflict`）。同じ名前を **別の** cBotで再利用できます。

JSONは **保存時に検証** されます。単一のフラットオブジェクトで、値がすべてスカラー（文字列/数値/ブール値）である必要があります。オブジェクト以外のルート、配列、ネストされたオブジェクト、`null` 値、または不正なJSONは拒否されます（ダイアログ内で明確なエラー、APIで `400 Bad Request`）。空のオブジェクト `{}` は許可され、「オーバーライドなし」を意味します。

## cTrader Console CLIの注記

バックテストは `--data-mode`（デフォルト `m1`）、`dd/MM/yyyy HH:mm` 形式の日付、`params.cbotset` JSON位置引数が必要です。`run` は `--data-dir` を拒否します（バックテストのみ）。`ContainerCommandHelpers` を参照してください。

## ノードとスケール

実行容量はノードエージェント（自己登録+ハートビート）を追加することでスケールします。[node discovery](../operations/node-discovery.md) および [scaling](../deployment/scaling.md) を参照してください。

## トレーディングアカウントが必要です

cBotを実行またはバックテストするには、cTrader トレーディングアカウントが接続する必要があります。**Trading accounts** に追加するまで、**Run New cBot** / **Backtest New cBot** ボタンは無効（ツールチップ付き）で、ページはアカウント設定にリンクするプロンプトを表示します。これにより、アカウントなしのbotから生の `stream connect failed` エラーが発生することがなくなります。
