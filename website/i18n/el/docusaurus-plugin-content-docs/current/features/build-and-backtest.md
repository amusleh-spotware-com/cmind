# Build & backtest cBots

Δημιουργία, εκτέλεση, backtest cTrader cBots (C# **και** Python, και τα δύο .NET) από το ενσωματωμένο
Monaco IDE, εκτέλεση στη επίσημη εικόνα `ghcr.io/spotware/ctrader-console`.

## Build

- Η σελίδα **Builder** φιλοξενεί τον επεξεργαστή Monaco· το `CBotBuilder` μεταγλωττίζει το έργο με
  `dotnet build` **σε αποδοχή κοντέινερ** (`AppOptions.BuildImage`, dir εργασίας bind-mount
  στο `/work`), έτσι ώστε τα ανέμπιστα user MSBuild targets να μην φθάνουν το host. Η ανάκτηση NuGet αποθηκεύεται
  σε κοινή κρυφή μνήμη μεταξύ δημιουργιών μέσω κοινού volume. Ο ιστότοπος web χρειάζεται πρόσβαση στη υποδοχή Docker.
- Τα templates εκκίνησης C# + Python βρίσκονται στο `src/Nodes/Builder/Templates/`.

## Run & backtest

- **Instances** = ιεραρχία κατάστασης TPH (`Run`/`Backtest` × `Pending`/`Scheduled`/`Starting`/
  `Running`/`Stopping`/`Stopped`/`Failed`). Η μετάβαση αντικαθιστά την οντότητα (αλλαγή id),
  η container id μεταφέρεται.
- Το `NodeScheduler` επιλέγει το λιγότερο φορτωμένο κατάλληλο Node· το `ContainerDispatcherFactory` δρομολογεί
  στο απομακρυσμένο Node HTTP agent ή στο τοπικό Docker dispatcher.
- Οι pollers ολοκλήρωσης συμφιλιώνουν τα containers που έχουν εξέλθει (backtest containers αυτο-έξοδος μέσω
  `--exit-on-stop`)· αναφορά παρούσα → ολοκληρώθηκε (αποθήκευση `ReportJson`), απουσία → απέτυχε.
- Τα ζωντανά αρχεία καταγραφής κοντέινερ ρέουν στο πρόγραμμα περιήγησης μέσω SignalR· οι καμπύλες ιστορικού backtest αναλύονται από
  την αναφορά + σχεδιάζονται.

## Backtest market data is cached per account

Το cTrader Console κατεβάζει ιστορικά δεδομένα τικ/μπάρας στο `--data-dir`. Αυτός ο κατάλογος είναι ένα
**σταθερό, μόνιμο cache με κλειδί το λογαριασμό διαπραγμάτευσης** (ο αριθμός του λογαριασμού) — bind-mounted από τον δίσκο του
Node στη δική του διαδρομή κοντέινερ (`/mnt/data`), ένα **ξεχωριστό, μη ένθετο mount** από το
per-instance dir εργασίας. Έτσι κάθε backtest στο ίδιο λογαριασμό **επαναχρησιμοποιεί** τα ήδη κατεβασμένα δεδομένα
αντί να τα κατεβάσει ξανά κάθε εκτέλεση. (Νωρίτερα ο
κατάλογος δεδομένων βρισκόταν κάτω από το per-instance dir εργασίας, του οποίου η id αλλάζει κάθε εκτέλεση, το οποίο ανάγκαζε μια νέα
λήψη κάθε backtest.) Ο εφήμερος per-instance dir εργασίας εξακολουθεί να κρατά τον αλγόριθμο, τις παραμέτρους, τον κωδικό πρόσβασης
και την αναφορά· το κοινό cache δεδομένων μετράται στη χρήση backtest-data ενός Node και καθαρίζεται από την
ενέργεια καθαρισμού Node.

## Backtest settings

Το διάλογο **Backtest** εκθέτει τις ρυθμίσεις backtest cTrader Console που μπορεί να ρυθμίσει ο χρήστης, έτσι δεν χρειάζεται ποτέ να
αγγίξετε μια γραμμή εντολών:

- **Symbol / Timeframe** — το timeframe είναι ένα **dropdown κάθε περιόδου cTrader** (`t1`…`t1000`,
  `m1`…`m45`, `h1`…`h12`, `D1`/`D2`/`D3`, `W1`, `Month1`, και τις περιόδους Renko/Range/Heikin), στο
  κανονικό casing της κονσόλας, έτσι ώστε πάντα να επιλέγετε ένα έγκυρο `--period`.
- **From / To** — το παράθυρο backtest (`--start` / `--end`).
- **Data mode** — ένα από τα τρία modes cTrader (`--data-mode`): **Tick data** (`tick`, ακριβή),
  **m1 bars** (`m1`, γρήγορα), ή **Open prices only** (`open`, πιο γρήγορα).
- **Starting balance** — προεπιλογή `10000` (`--balance`). Ένα **0 balance δεν τοποθετεί trades και κάνει το
  cTrader να εκπέμπει ένα κενό report που στη συνέχεια παρουσιάζει σφάλμα** ("Message expected"), έτσι ένα μη-μηδενικό υπόλοιπο είναι
  πάντα αποστέλλεται.
- **Commission** και **Spread** — `--commission` / `--spread` (spread σε pips).

Ο κατάλογος δεδομένων (`--data-file` / `--data-dir`) διαχειρίζεται από την ίδια την εφαρμογή (ένα per-account cache, δείτε
παραπάνω), όχι εκτεθειμένο στο διάλογο.

## Instance detail page

Το άνοιγμα μιας instance (`/instance/{id}`) εμφανίζει την κατάστασή της ζωντανή, τα αρχεία καταγραφής και — για ένα backtest — το
καμπύλη ιστορικού. Ο **τίτλος καρτέλας του προγράμματος περιήγησης** αντικατοπτρίζει τη συγκεκριμένη instance (**όνομα cBot · τύπος · σύμβολο**, π.χ.
`TrendBot · Backtest · EURUSD`) έτσι ώστε μια καρτέλα ζωντανής εκτέλεσης και μια καρτέλα backtest να είναι διακριτές με μια ματιά.
Μια εκτέλεση και ένα backtest του ίδιου cBot ορίζονται ως ξεχωριστές **lineages** (μια σταθερή lineage id που μεταφέρεται
σε όλες τις μεταβάσεις κατάστασης), έτσι η σελίδα ακολουθεί ακριβώς μια instance και δεν αναμιγνύει ποτέ δεδομένα εκτέλεσης με ένα
backtest.

## Instance lifecycle controls

Κάθε σειρά instance (και η σελίδα λεπτομερειών της) έχει στοιχεία ελέγχου που ταιριάζουν στην κατάσταση. Μια **ενεργή** instance εμφανίζει
**Stop**· μια **τερματική** (Stopped / Completed / Failed) εμφανίζει **Start (▶)** για την επανεκκίνηση με
το ίδιο cBot, λογαριασμό, σύμβολο, χρονικό πλαίσιο, ParamSet και εικόνα (μια εκτέλεση επανεκκινείται ως εκτέλεση, ένα
backtest ως backtest). Κάνοντας κλικ Stop εμφανίζεται μια ειδοποίηση "Διακοπή…" και απενεργοποιεί το εικονίδιο μέχρι να λυθεί, και μια νέα δημιουργία
εκτέλεση εμφανίζεται στη λίστα αμέσως — χωρίς επαναφόρτωση σελίδας.

Τα αρχεία καταγραφής κονσόλας **παραμένουν όταν μια instance τερματίζει** — για μια εκτέλεση (κατά την διακοπή) και για ένα
**backtest** (κατά την ολοκλήρωση) ομοίως — έτσι τα αρχεία καταγραφής της τελευταίας εκτέλεσης παραμένουν δείγματα στη σελίδα λεπτομερειών και,
μέσω της γραμμής εργαλείων αρχείων καταγραφής, **αντιγράφονται στο πρόχειρο** (εικονίδιο Αντιγραφή αρχείων καταγραφής) ή **λήψη** (Λήψη αρχείων καταγραφής
εικονίδιο) ακόμα και μετά την αναγνώριση του κοντέινερ. Και τα δύο δρουν στο πλήρες αρχείο καταγραφής κονσόλας της instance, όχι μόνο στην
ουρά της οθόνης.

Ένα ανεβασμένο `.algo` δεν κατασκευάστηκε ποτέ εδώ, έτσι η στήλη **Last Build** στη σελίδα cBots αφήνεται
κενή (δείχνει μια ώρα δημιουργίας μόνο για cBots που κατασκευάζετε στο πρόγραμμα περιήγησης).

## Edit & re-run a stopped instance

Μια **διακοπημένη** instance (εκτέλεση ή backtest) έχει ένα στοιχείο ελέγχου **Edit** — ένα εικονίδιο στη σειρά της στη λίστα **και**
δίπλα στο Start/Stop στη σελίδα λεπτομερειών — που ανοίγει ένα διάλογο **προϊσχυρισμένο** με τη τρέχουσα διαμόρφωσή του.
Μπορείτε να αλλάξετε το **λογαριασμό διαπραγμάτευσης, σύμβολο, χρονικό πλαίσιο, ParamSet και ετικέτα εικόνας** (και, για ένα
backtest, το **παράθυρο και όλες τις ρυθμίσεις backtest** παραπάνω), στη συνέχεια **Αποθήκευση & Εκκίνηση** επανεκκινεί με
τις νέες ρυθμίσεις (αντικατάσταση της διακοπημένης instance). Το στοιχείο ελέγχου είναι **απενεργοποιημένο ενώ η instance είναι ενεργή** —
μόνο μια διακοπημένη instance μπορεί να επεξεργαστεί.

## Run from the code editor

Κάνοντας κλικ **Run** στον επεξεργαστή κώδικα ανοίγει ένα διάλογο αντί να ξεκινήσει μια τυφλή, hard-coded εκτέλεση:

- **Trading account** (απαιτούμενο) — ο λογαριασμός cTrader στον οποίο συνδέεται το cBot.
- **Parameter set** (προαιρετικό) — επιλέξτε ένα υπάρχον σύνολο, ή αφήστε το κενό για εκτέλεση με το cBot
  **προεπιλεγμένες τιμές παραμέτρων**. Ένα **+** κουμπί δίπλα στο σelector δημιουργεί ένα νέο ParamSet
  inline (δείτε παρακάτω) και το επιλέγει.
- **Symbol / Timeframe** προεπιλογή `EURUSD` / `h1` και μπορούν να αλλάξουν· **Ακύρωση** ή **Εκτέλεση**.

Με κλικ **Εκτέλεση** ο επεξεργαστής αποθηκεύει + κατασκευάζει την τρέχουσα πηγή, ξεκινά την instance στο επιλεγμένο λογαριασμό
με τις επιλεγμένες παραμέτρους, στη συνέχεια ουρά τα ζωντανά αρχεία καταγραφής κοντέινερ. (Το ρεύμα αρχείων καταγραφής προωθεί το
cookie ελέγχου ταυτότητας του χρήστη που είναι συνδεδεμένος στο hub SignalR `/hubs/logs`, έτσι συνδέεται αντί να αποτυγχάνει με
`Invalid negotiation response received`.)

## Parameter sets

Ένα **ParamSet** είναι ένα ονοματισμένο, επαναχρησιμοποιήσιμο σύνολο παραμέτρων override του cBot αποθηκευμένο ως ένα επίπεδο JSON
αντικείμενο που χαρτογραφεί κάθε όνομα παραμέτρου σε μια κλιμακωτή τιμή, π.χ. `{"Period": 14, "Label": "trend"}`. Κατά τη διάρκεια του
εκτέλεσης/backtest μετατρέπεται στο αρχείο cTrader `params.cbotset`
(`{ "Parameters": { … } }`). Μπορείτε να δημιουργήσετε/επεξεργαστείτε ένα σύνολο ως raw JSON από το διάλογο **Parameter sets** του cBot ή inline από το διάλογο Εκτέλεσης.

Κάθε ParamSet **ανήκει σε ένα cBot**: το διάλογο New Parameter Set καταγράφει όλα τα cBots σας και πρέπει
**να επιλέξετε ένα** — η δημιουργία αποκλείεται έως ότου επιλεγεί ένα cBot. Το **όνομα ενός συνόλου είναι μοναδικό ανά cBot**:
η δημιουργία ή η μετονομασία ενός συνόλου σε ένα όνομα που ένα άλλο σύνολο του ίδιου cBot χρησιμοποιεί ήδη απορρίπτεται (ένα σαφές
σφάλμα στο διάλογο, `409 Conflict` στο API). Το ίδιο όνομα μπορεί να χρησιμοποιηθεί ξανά σε ένα **διαφορετικό** cBot.

Το JSON **επικυρώνεται** κατά την αποθήκευση: πρέπει να είναι ένα μόνο επίπεδο αντικείμενο του οποίου οι τιμές είναι όλες κλιμακωτές
(string / number / bool). Ένα μη-αντικείμενο root, ένας πίνακας, ένα ένθετο αντικείμενο, ένα `null` τιμή, ή κακογραμμένο
JSON απορρίπτεται (ένα σαφές σφάλμα στο διάλογο, `400 Bad Request` στο API). Ένα κενό αντικείμενο `{}`
επιτρέπεται και σημαίνει "χωρίς overrides".

## cTrader Console CLI notes

Τα backtests χρειάζονται `--data-mode` (προεπιλογή `m1`), ημερομηνίες ως `dd/MM/yyyy HH:mm`, και
`params.cbotset` JSON θέση arg· `run` απορρίπτει `--data-dir` (backtest-only). Δείτε
`ContainerCommandHelpers`.

## Nodes & scale

Η ικανότητα εκτέλεσης κλιμακώνεται με την προσθήκη Node agents (αυτο-εγγραφή + ανάπαλα). Δείτε
[ανακάλυψη Node](../operations/node-discovery.md) και [κλίμακα](../deployment/scaling.md).

## A trading account is required

Η εκτέλεση ή backtest ενός cBot απαιτεί έναν λογαριασμό διαπραγμάτευσης cTrader για σύνδεση. Έως ότου προσθέσετε ένα κάτω από
**Trading accounts**, τα κουμπιά **Run New cBot** / **Backtest New cBot** είναι απενεργοποιημένα (με ένα
tooltip) και η σελίδα εμφανίζει μια προειδοποίηση που συνδέει στη ρύθμιση λογαριασμού — δεν χτυπάτε αυτό ποτέ
`stream connect failed` σφάλμα από ένα bot χωρίς λογαριασμό.
