# cBotのビルド＆バックテスト

ブラウザ内Monaco IDEからcTrader cBot（C# **および** Python、両方.NET）をビルド、実行、バックテストします。公式`ghcr.io/spotware/ctrader-console`イメージで実行できます。

## ビルド

- **Builder** ページはMonaco editorをホストし、`CBotBuilder`は`dotnet build`を**使い捨てコンテナ内**で実行します（`AppOptions.BuildImage`、作業ディレクトリは`/work`にバインドマウント）。信頼できないユーザーのMSBuildターゲットがホストに到達しません。NuGetリストアは共有ボリュームを介してビルド間でキャッシュされます。WebホストはDockerソケットアクセスが必要です。
- C# + Python スターターテンプレートは `src/Nodes/Builder/Templates/` にあります。

## 実行＆バックテスト

- **Instances** = TPH状態階層（`Run`/`Backtest` × `Pending`/`Scheduled`/`Starting`/`Running`/`Stopping`/`Stopped`/`Failed`）。状態遷移はエンティティを置き換えます（ID変更）。コンテナIDは引き継がれます。
- `NodeScheduler`は最も負荷が低い適格なノードを選択します。`ContainerDispatcherFactory`はリモートノードのHTTPエージェントまたはローカルDockerディスパッチャーにルーティングします。
- 完了ポーラーは終了したコンテナを調整します（バックテストコンテナは`--exit-on-stop`で自動終了）。レポートが存在する→完了として記録（`ReportJson`を保存）、存在しない→失敗とします。
- ライブコンテナログはSignalRを介してブラウザにストリーミングされます。バックテストエクイティカーブはレポートから解析され、グラフ化されます。

## バックテスト市場データはアカウントごとにキャッシュされます

cTrader Consoleは履歴ティック/バーデータを`--data-dir`にダウンロードします。このディレクトリは**取引アカウント（そのアカウント番号）をキーとした安定した永続キャッシュ**です。ノードのディスクからそのコンテナパス（`/mnt/data`）にバインドマウントされます。これは、インスタンスごとの作業ディレクトリから**分離した、ネストされていないマウント**です。したがって、同じアカウントでの各バックテストは、すでにダウンロード済みのデータを**再利用**し、実行のたびに新しくダウンロードする必要がありません。（以前、データディレクトリはインスタンスごとの作業ディレクトリ配下にありましたが、その ID は実行するたびに変わるため、バックテストのたびに新しいダウンロードを強制しました。）エフェメラルなインスタンスごとの作業ディレクトリはアルゴ、パラメータ、パスワードとレポートをまだ保持しており、共有データキャッシュはノードのバックテストデータ使用量でカウントされ、ノードクリーンアクションによってクリアされます。

## バックテスト設定

**Backtest** ダイアログはユーザーが調整可能なcTrader Consoleバックテスト設定を公開するため、コマンドラインに触れる必要はありません：

- **Symbol / Timeframe** — タイムフレームはcTraderのあらゆるピリオドのドロップダウンです（`t1`…`t1000`、`m1`…`m45`、`h1`…`h12`、`D1`/`D2`/`D3`、`W1`、`Month1`、およびRenko/Range/Heikinピリオド）。コンソールの正規のケーシングを使用するため、常に有効な`--period`を選択できます。
- **From / To** — バックテストウィンドウ（`--start` / `--end`）。
- **Data mode** — 3つのcTraderモードのいずれか（`--data-mode`）：**Tick data**（`tick`、正確）、**m1 bars**（`m1`、高速）、または**Open prices only**（`open`、最速）。
- **Starting balance** — デフォルトは`10000`（`--balance`）。**0残高は取引をしないようにし、cTraderに空のレポートを発行させます。その後クラッシュします**（"Message expected"）。そのため、常に0以外の残高が送信されます。
- **Commission** および **Spread** — `--commission` / `--spread`（スプレッドはピップス単位）。

データディレクトリ（`--data-file` / `--data-dir`）はアプリ自体で管理されます（アカウントごとのキャッシュ、上記参照）。ダイアログには公開されていません。

## インスタンス詳細ページ

インスタンス（`/instance/{id}`）を開くと、そのライブステータス、ログ、およびバックテストの場合はエクイティカーブが表示されます。**ブラウザタブタイトル**は特定のインスタンス（**cBot名·種類·シンボル**、例：`TrendBot · Backtest · EURUSD`）を反映するため、ライブ実行タブとバックテストタブは一目で区別できます。同じcBotのランとバックテストは異なる**リネージ**として追跡されます（状態遷移を通じて実行されるリネージID）。ページは正確に1つのインスタンスを追従し、ランのデータとバックテストのデータが混在することはありません。

## インスタンスライフサイクルコントロール

各インスタンス行（とその詳細ページ）には状態に適したコントロールがあります。**アクティブな**インスタンスは**Stop**を表示します。**ターミナル**状態（Stopped / Completed / Failed）は**Start（▶）**を表示して、同じcBot、アカウント、シンボル、タイムフレーム、パラメータセット、イメージで再実行できます（ランはランとして再実行、バックテストはバックテストとして再実行）。Stopをクリックすると「Stopping…」という通知が表示され、解決するまでアイコンが無効になります。新しく作成されたランはリストに即座に表示されます。ページリロードは不要です。

コンソールログは**インスタンスが終了したときに永続化**されます。ラン（停止時）とバックテスト（完了時）の両方。したがって、最後のランのログは詳細ページで表示可能なままで、ログツールバー経由で**クリップボードにコピー**（Copy logsアイコン）または**ダウンロード**（Download logsアイコン）できます。コンテナが削除された後も同様です。どちらもインスタンスの完全なコンソールログに作用し、画面上のテール部分だけではありません。

アップロードされた`.algo`がここでビルドされたことはないため、cBotsページの**Last Build**列は空白のままです（ブラウザでビルドしたcBotのみビルド時刻を表示します）。

## 停止したインスタンスを編集＆再実行

**停止した**インスタンス（ランまたはバックテスト）は**Edit**コントロール（リスト内のその行のアイコン**と**詳細ページのStart/Stopの隣）があり、ダイアログを開きます。これは現在の設定で**事前入力**されています。**取引アカウント、シンボル、タイムフレーム、パラメータセット、およびイメージタグ**を変更できます（バックテストの場合、**ウィンドウとすべてのバックテスト設定**も）。その後**Save & start**で新しい設定で再実行します（停止したインスタンスを置き換えます）。このコントロールは**インスタンスがアクティブな間は無効**です。停止したインスタンスのみ編集可能です。

## コードエディターから実行

コードエディターで**Run**をクリックすると、ブラインド、ハードコードされたランを発火する代わりにダイアログが開きます：

- **Trading account**（必須） — cBotが接続するcTraderアカウント。
- **Parameter set**（オプション） — 既存のセットを選択するか、空のままにして**デフォルトパラメータ値**でcBotを実行します。セレクターの隣の**+**ボタンは新しいパラメータセットをインラインで作成し（下記参照）、それを選択します。
- **Symbol / Timeframe**はデフォルトで`EURUSD` / `h1`。変更可能。**Cancel** または **Run**。

**Run**をクリックするとエディターは現在のソースを保存＆ビルドし、選択されたアカウントで選択されたパラメータでインスタンスを開始し、ライブコンテナログをテーリングします。（ログストリームは署名されたユーザーの認証クッキーを`/hubs/logs` SignalR hubに転送するため、`Invalid negotiation response received`で失敗する代わりに接続します。）

## パラメータセット

**parameter set**は名前が付けられた、再利用可能なcBotパラメータオーバーライドのセットです。フラットJSONオブジェクトとして保存されます。各パラメータ名をスカラー値にマッピングします。例：`{"Period": 14, "Label": "trend"}`。実行/バックテスト時、それはcTrader `params.cbotset`ファイルに変換されます（`{ "Parameters": { … } }`）。cBotの**Parameter sets**ダイアログからパラメータセットをRaw JSONとして作成/編集するか、Runダイアログからインラインで作成できます。

各パラメータセットは**cBotに属する**：新しいパラメータセットダイアログはすべてのcBotをリストし、**1つを選択する必要があります**。作成はcBotが選択されるまでブロックされます。セットの**名前はcBotごとに一意**です。セットを、同じcBotの別のセットがすでに使用している名前に作成またはリネームするのは拒否されます（ダイアログでの明確なエラー、APIでの`409 Conflict`）。同じ名前は**異なる** cBotで再利用できます。

JSONは**保存時に検証**されます。単一のフラットオブジェクトである必要があり、その値はすべてスカラー（文字列/数値/ブール値）です。非オブジェクトルート、配列、ネストされたオブジェクト、`null`値、または形式が正しくないJSONは拒否されます（ダイアログでの明確なエラー、APIでの`400 Bad Request`）。空のオブジェクト`{}`は許可されており、「オーバーライドなし」を意味します。

## cTrader Console CLIノート

バックテストには`--data-mode`（デフォルト`m1`）、日付を`dd/MM/yyyy HH:mm`形式、および`params.cbotset` JSONポジション引数が必要です。`run`は`--data-dir`を拒否します（バックテストのみ）。`ContainerCommandHelpers`を参照してください。

## ノード＆スケール

実行容量はノードエージェントを追加することでスケールします（自動登録＋ハートビート）。[ノード検出](../operations/node-discovery.md)と[スケーリング](../deployment/scaling.md)を参照してください。

## 取引アカウントが必要です

cBotを実行またはバックテストするには、接続するcTrader取引アカウントが必要です。**Trading accounts**に追加するまで、**Run New cBot** / **Backtest New cBot**ボタンは無効化されます（ツールチップ付き）。ページはアカウント設定へのリンク付きプロンプトを表示します。生のアカウントなしのbot から`stream connect failed`エラーが出ることはなくなります。
