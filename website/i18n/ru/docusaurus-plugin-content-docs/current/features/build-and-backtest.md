# Build & backtest cBots

Создавайте, запускайте и тестируйте cBot cTrader (C# **и** Python, оба .NET) из браузерного редактора Monaco IDE, запускайте на официальном образе `ghcr.io/spotware/ctrader-console`.

## Build

- **Builder** страница хостит редактор Monaco; `CBotBuilder` компилирует проект с помощью
  `dotnet build` **в одноразовом контейнере** (`AppOptions.BuildImage`, рабочий каталог привязан
  к `/work`), поэтому ненадежные цели MSBuild пользователя не могут достичь хоста. Восстановление NuGet кешируется
  между сборками через общий том. Веб-хост нуждается в доступе к сокету Docker.
- Стартовые шаблоны C# и Python находятся в `src/Nodes/Builder/Templates/`.

## Run & backtest

- **Instances** = иерархия состояния TPH (`Run`/`Backtest` × `Pending`/`Scheduled`/`Starting`/
  `Running`/`Stopping`/`Stopped`/`Failed`). Переход заменяет сущность (изменение id),
  id контейнера переносится.
- `NodeScheduler` выбирает наименее загруженный подходящий узел; `ContainerDispatcherFactory` маршрутизирует на
  удаленный HTTP агент узла или локальный диспетчер Docker.
- Полиры завершения согласовывают вышедшие контейнеры (контейнеры backtest самостоятельно выходят через
  `--exit-on-stop`); отчет присутствует → завершено (сохраняет `ReportJson`), отсутствует → не выполнено.
- Живые логи контейнера потоком передаются в браузер через SignalR; кривые капитализации backtest анализируются из
  отчета и отображаются в диаграмме.

## Backtest market data is cached per account

Консоль cTrader загружает исторические данные тиков/баров в свою папку `--data-dir`. Эта папка
**стабильный, постоянный кеш, привязанный к торговому счету** (его номеру счета) — привязана с диска узла
в его собственный путь контейнера (`/mnt/data`), **отдельное, не вложенное крепление** из
папки работы для каждого экземпляра. Поэтому каждый backtest на одном счете **переиспользует** уже загруженные данные
вместо их повторной загрузки при каждом запуске. (Ранее
папка данных находилась под папкой работы для каждого экземпляра, чей id меняется при каждом запуске, что вынуждало
свежую загрузку при каждом backtest.) Временная папка работы для каждого экземпляра по-прежнему содержит algo, параметры, пароль
и отчет; общий кеш данных учитывается в использовании backtest-data узлом и очищается действием
node-clean.

## Backtest settings

Диалог **Backtest** открывает настраиваемые пользователем параметры backtest консоли cTrader, поэтому вам никогда не потребуется
касаться командной строки:

- **Symbol / Timeframe** — временной фрейм — **выпадающий список каждого периода cTrader** (`t1`…`t1000`,
  `m1`…`m45`, `h1`…`h12`, `D1`/`D2`/`D3`, `W1`, `Month1` и периоды Renko/Range/Heikin), в
  каноническом формате консоли, поэтому вы всегда выбираете действительный `--period`.
- **From / To** — окно backtest (`--start` / `--end`).
- **Data mode** — один из трех режимов cTrader (`--data-mode`): **Tick data** (`tick`, точный),
  **m1 bars** (`m1`, быстрый), или **Open prices only** (`open`, самый быстрый).
- **Starting balance** — по умолчанию `10000` (`--balance`). **Баланс 0 не размещает никаких сделок и заставляет
  cTrader выдать пустой отчет, который затем вызывает ошибку** ("Message expected"), поэтому ненулевой баланс
  всегда отправляется.
- **Commission** — `--commission`.
- **Spread** — `--spread`, **числовое поле в пипсах, которое не может быть ниже 0**. Оно **скрыто в режиме Tick
  data**, где cTrader вычисляет спред из самих данных тиков (не отправляется `--spread`).

Папка данных (`--data-file` / `--data-dir`) управляется самим приложением (кеш для каждого счета, см.
выше), не открывается в диалоге.

:::note cTrader crashes on an empty backtest
Если backtest не выдает **никаких результатов** — без сделок или без данных рынка для выбранных дат/символа —
собственный модуль записи отчетов консоли cTrader выбрасывает `Message expected` и выходит без отчета. Приложение не может
исправить эту ошибку upstream, но оно ее обнаруживает и помечает экземпляр как **Failed** с понятной причиной
("no backtest results for the selected range…") вместо сырого трассировки стека. Выберите более широкий диапазон дат
с доступными данными рынка и повторите попытку.
:::

## Instance detail page

Открытие экземпляра (`/instance/{id}`) показывает его живой статус, логи и — для backtest — кривую капитализации.
Заголовок **браузерной вкладки** отражает конкретный экземпляр (**имя cBot · тип · символ**, например
`TrendBot · Backtest · EURUSD`), поэтому вкладка live-run и вкладка backtest различимы с первого взгляда.
Запуск и backtest одного cBot отслеживаются как отдельные **lineages** (стабильный id lineage переносится
через переходы состояния), поэтому страница следует ровно одному экземпляру и никогда не смешивает данные запуска с данными
backtest.

## Instance lifecycle controls

Каждая строка экземпляра (и его страница деталей) имеет управления, соответствующие состоянию. **Активный** экземпляр показывает
**Stop**; **терминальный** (Stopped / Completed / Failed) показывает **Start (▶)** для повторного запуска с
тем же cBot, счетом, символом, временным фреймом, набором параметров и образом (запуск перезапускается как запуск, backtest как backtest). Нажатие Stop показывает уведомление "Stopping…" и отключает значок до
его разрешения, и новый запущенный экземпляр появляется в списке немедленно — без перезагрузки страницы.

Логи консоли **сохраняются, когда экземпляр завершается** — для запуска (при остановке) и для
**backtest** (при завершении) одинаково — поэтому логи последнего запуска остаются доступны на странице деталей и,
через панель инструментов логов, **скопированы в буфер обмена** (значок Copy logs) или **загружены** (значок Download logs)
даже после исчезновения контейнера. Оба действуют на полный консольный лог экземпляра, а не только
на видимую часть.

Загруженный `.algo` никогда не был собран здесь, поэтому его столбец **Last Build** на странице cBots оставлен
пустым (он показывает время сборки только для cBots, которые вы собираете в браузере).

## Edit & re-run a stopped instance

**Остановленный** экземпляр (запуск или backtest) имеет **Edit** управление — значок на его строке в списке **и**
рядом с Start/Stop на его странице деталей — который открывает диалог **предзаполненный** его текущей конфигурацией.
Вы можете изменить **торговый счет, символ, временной фрейм, набор параметров и тег образа** (и, для
backtest, **окно и все параметры backtest** выше), затем **Save & start** перезапускает его с
новыми параметрами (заменяя остановленный экземпляр). Управление **отключено, пока экземпляр активен** —
только остановленный экземпляр может быть отредактирован.

## Run from the code editor

Нажатие **Run** в редакторе кода открывает диалог вместо слепого, жестко закодированного запуска:

- **Trading account** (требуется) — счет cTrader, к которому подключается cBot.
- **Parameter set** (необязательно) — выберите существующий набор или оставьте его пустым для запуска с
  **значениями параметров cBot по умолчанию**. Кнопка **+** рядом с селектором создает новый набор параметров
  встроенно (см. ниже) и выбирает его.
- **Symbol / Timeframe** по умолчанию установлены на `EURUSD` / `h1` и могут быть изменены; **Cancel** или **Run**.

При нажатии **Run** редактор сохраняет + собирает текущий источник, запускает экземпляр на выбранном счете
с выбранными параметрами, затем отслеживает живые логи контейнера. (Поток логов пересылает
cookie аутентификации вошедшего пользователя на хаб SignalR `/hubs/logs`, поэтому он подключается вместо того, чтобы не выполняться с
`Invalid negotiation response received`.)

## Parameter sets

**Parameter set** — это именованный, переиспользуемый набор переопределений параметров cBot, сохраненный как плоский объект JSON
отображающий каждое имя параметра на скалярное значение, например `{"Period": 14, "Label": "trend"}`. При
запуске/backtest времени он преобразуется в файл `params.cbotset` cTrader
(`{ "Parameters": { … } }`). Вы можете создавать/редактировать набор как сырой JSON из диалога **Parameter
sets** cBot или встроенно из диалога Run.

Каждый набор параметров **принадлежит cBot**: диалог New Parameter Set перечисляет все ваши cBots и вы
**должны выбрать один** — создание блокируется до выбора cBot. **Имя набора уникально в пределах cBot**:
создание или переименование набора на имя, которое другой набор того же cBot уже использует, отклоняется (понятная
ошибка в диалоге, `409 Conflict` в API). То же имя может быть переиспользовано на **другом** cBot.

JSON **валидируется** при сохранении: это должен быть один плоский объект, чьи значения все скаляры
(string / number / bool). Корневой не-объект, массив, вложенный объект, значение `null` или неправильный
JSON отклоняются (понятная ошибка в диалоге, `400 Bad Request` в API). Пустой объект `{}`
разрешен и означает "нет переопределений".

## cTrader Console CLI notes

Backtests нужен `--data-mode` (по умолчанию `m1`), даты как `dd/MM/yyyy HH:mm`, и
JSON аргумент `params.cbotset` позиционный; `run` отклоняет `--data-dir` (только для backtest). Смотрите
`ContainerCommandHelpers`.

## Nodes & scale

Емкость выполнения масштабируется добавлением агентов узлов (самостоятельная регистрация + heartbeat). Смотрите
[node discovery](../operations/node-discovery.md) и [scaling](../deployment/scaling.md).
## A trading account is required

Запуск или backtest cBot нуждается в торговом счете cTrader для подключения. До тех пор, пока вы не добавите его в
**Trading accounts**, кнопки **Run New cBot** / **Backtest New cBot** отключены (с
подсказкой) и страница показывает подсказку со ссылкой на настройку счета — вы больше не попадаете на сырую
ошибку `stream connect failed` от бота без счета.
