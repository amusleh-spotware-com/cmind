# Open API 토큰 생명 주기

cTrader의 Open API는 **한 번에 cTrader ID (cID)당 유효한 액세스 토큰 하나**를 허용합니다. 새 토큰이 발급되는 순간 — 예약된 새로고침 또는 사용자가 동일 cID에서 다른 계정을 연결할 때 재인증 — 이전 액세스 토큰은 무효화됩니다. 원격 노드에서 실행 중인 복사 엔진이 이제 죽은 토큰을 보유하고 있으므로 새 토큰은 라이브 연결을 떨어뜨리지 않고 도달해야 합니다.

## 모델

- **`OpenApiAuthorization`**은 cID의 암호화된 액세스 + 새로고침 토큰을 보유하는 집계입니다. `(UserId, CtidUserId)`의 고유 인덱스는 **사용자당 cID당 정확히 하나의 인증**을 적용합니다.
- **`TokenVersion`** — 토큰이 회전할 때마다 범프되는 단조 카운터 (`Refresh()`, 동일 cID에서 다른 계정이 링크될 때 재인증 경로도 포함). 단일 유효 토큰 규칙의 버전 마커이며 두 토큰 문자열이 우연히 충돌하더라도 실행 호스트가 변경을 감지하는 데 사용하는 것입니다.
- 토큰은 `ISecretProtector`를 통해 저장 시 암호화됩니다 (`EncryptionPurposes.OpenApiAccessToken` / `OpenApiRefreshToken`). 절대 평문으로 기록되거나 저장되지 않습니다.

## 전파 (우아한 제자리 스왑)

1. 토큰 회전 → 새 토큰 + 범프된 `TokenVersion`이 유지됩니다.
2. 호스팅 노드의 `CopyEngineSupervisor`는 각 조정 사이클에서 계획을 다시 읽고 **토큰 서명** (액세스 토큰 + 버전)을 계산합니다. 변경은 회전을 의미합니다.
3. 호스트를 해체하고 재시작하는 대신 (마스터의 실행 스트림을 드롭하는), 감독자는 **실행 호스트에 새 토큰을 푸시합니다**.
4. 호스트는 **기존 소켓**에서 영향을 받는 계정을 다시 인증합니다 (`ProtoOAAccountAuthReq` 다시) `SwapAccessTokenAsync`를 통해, 그 다음 가벼운 조정을 수행합니다. 오래된 토큰은 죽습니다; 복사 스트림은 절대 중지되지 않습니다.

이것이 크로스 cID 경우를 안전하게 하는 것입니다: 사용자가 중간 실행 중에 동일 cID에서 두 번째 계정을 추가하면 오래된 토큰을 무효화하고 실행 복사 프로필은 새로운 토큰으로 계속됩니다.

## 새로고침

`OpenApiTokenRefreshService` (백그라운드)는 만료 전에 인증을 사전에 새로고칩니다; `OpenApiAuthorization.IsExpiring(threshold, now)`는 이를 게이트합니다. cTrader는 모든 새로고침에서 **새로고침** 토큰을 회전하므로 새 새로고침 토큰은 즉시 유지됩니다; 유지할 수 없는 읽기 전용 캐시는 자체 무효화됩니다 (클러스터 내 테스트 Job과 관련, 비밀의 쓰기 가능 복사본을 마운트).

### 실패 에스컬레이션

실패한 새로고침은 조용하지 않습니다. `OpenApiAuthorization.MarkRefreshFailed(reason, now, criticalWindow)`는 `RefreshFailedAt`을 기록하고, `ConsecutiveRefreshFailures`를 증가시키고, 항상 `AccessTokenRefreshFailed` (경고)를 발생시킵니다. 토큰이 이제 `App:OpenApi:TokenRefreshCriticalWindow` (기본값 6h) 내 만료이고 새로고침이 여전히 실패하면 **한 번** `AccessTokenRefreshCritical` 도메인 이벤트 + `Critical` 로그로 에스컬레이션하여 소유자가 복사/프로퍼 펌 작업이 토큰을 잃기 전에 재인증할 수 있습니다. 실패 카운터 및 에스컬레이션 래치는 다음 성공적인 `Refresh`에서 재설정됩니다. 서비스는 매 `TokenRefreshInterval`마다 재시도를 계속하므로 공급자/유지보수 중단은 새로고침 엔드포인트가 반환할 때 자체 복구됩니다.

## 무효화 경고 및 자동 복구 (M1)

cID의 부분/다시 인증은 실행 복사 호스트가 여전히 보유하고 있는 토큰을 무효화합니다. 거래 호출이 `OpenApiErrorKind.TokenInvalid`로 거절하면 호스트는 서로 다른 **`CopyTokenInvalidated`** 경고 (로그 1078)를 발생시킵니다 — 제네릭 실패가 아님 — 따라서 알림 채널은 토큰이 주의가 필요한지 알 수 있습니다. 복구는 자동입니다: 감독자는 각 사이클에서 인증을 다시 읽고, 새로고침된 토큰이 토큰 서명을 변경하면 실행 호스트로 **제자리 스왑**하도록 푸시합니다 — 복사는 수동 재추가 없이 재개됩니다. `NotLinkable` 프로필 (토큰/인증이 임시로 해석 불가능)은 마찬가지로 감독자 사이클마다 재평가되고 해당 계획이 다시 빌드되는 순간 호스팅됩니다.

## 호스트 생존성 감시 개(M2)

감독자는 각 호스팅된 프로필의 실행 작업을 봅니다. 호스트가 프로필이 여전히 이 노드에 할당된 상태에서 종료되거나 오류가 발생하면 감시 개는 다음 사이클에서 취소하고 **재시작**합니다 (로그 `CopyHostRestarted`). 따라서 끼인 호스트는 수동 재시작이 필요하지 않고 자체 복구되고 — 하나의 프로필의 실패는 다른 것을 절대 지연시키지 않습니다 (프로필 격리).

## 테스트

- **단위** — `TokenVersion`은 `Refresh`에서 범프; 호스트는 재시작 없이 제자리 스왑을 수행; 크로스 cID 무효화는 소스 및 목적지 토큰을 스왑; **무효화된 목적지 토큰은 `CopyTokenInvalidated`를 발생시키고 다음 토큰 푸시에서 자동 복구됩니다** (M1); 감시 개 `IsHostDead` 결정은 완료/오류 호스트를 재시작하고 재할당된 프로필을 혼자 둡니다 (M2).
- **통합** — `TokenVersion`은 실제 Postgres에서 EF를 통해 유지 + 증가; 토큰 서명은 버전 범프에서 변경됩니다. 문자열이 변경되지 않았더라도.
