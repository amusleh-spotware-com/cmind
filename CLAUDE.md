# CLAUDE.md

Guidance for future Claude Code sessions working on this repository.

## What this repo is

A multi-tenant Blazor Server + Minimal API application that lets users build, run, and
backtest cTrader cBots through the official `ghcr.io/spotware/ctrader-console` Docker image,
spread across SSH-accessible nodes. Includes an in-browser code editor (Monaco) that
builds cBot projects (C# and Python, both .NET-based) locally on the web host using the
`dotnet build` toolchain.

The entire project was generated by Claude Code as an experiment. There is **no human-written
code** in this repository.

## Conventions

- **C# version**: latest (analyzers: `latest-default`). `TreatWarningsAsErrors=true`. There
  is intentionally no `NoWarn` list — fix real warnings instead of hiding them.
- **Configuration**: every config value goes through `IOptionsMonitor<CtwOptions>`. There are
  no `cfg["Key"]` reads in business code anywhere. `CtwOptions` is an immutable record bound
  to the `Ctw` section.
- **Logging**: use the source-generated `LogMessages` partial class
  (`src/Core/Logging/LogMessages.cs`) — extension methods on `ILogger`. Never call
  `ILogger.LogInformation(...)` directly with interpolated/templated strings inline.
- **Constants**: every magic string lives in `src/Core/Constants/AppConstants.cs` (or another
  file in `Core/Constants/`). Endpoint paths, policy names, claim types, encryption purposes,
  Docker image names, CLI flags — all constants.
- **Soft delete**: every entity inherits from `AuditedEntity` which implements `ISoftDeletable`
  (`IsDeleted`, `DeletedAt`). `CtwDbContext.OnModelCreating` adds a global query filter that
  hides soft-deleted rows. `SaveChanges` intercepts `EntityState.Deleted` and converts it to
  `Modified` with the soft-delete flags set.
- **Strong types**: domain identifiers and value objects live in `src/Core/StrongIds.cs`
  (`UserId`, `NodeId`, ..., `Email`, `Symbol`, `Timeframe`, `DockerImageTag`). Note: entity
  classes currently keep `Guid` properties for EF mapping simplicity — full strong-typed
  EF integration via value converters is a follow-up.
- **Encryption**: never log or store secrets in plaintext. Use `ISecretProtector` with the
  purpose strings from `EncryptionPurposes`. Data Protection key ring is encrypted with a
  PFX certificate provided via base64 environment variable.
- **Health checks**: `AddHealthChecks().AddNpgSql(...)` in `Web/Program.cs`. Mapped in
  Development by `MapDefaultEndpoints`.

## Notable design decisions

- Builder runs **locally on the web host**, not on remote nodes. Web container needs Docker
  socket access (`-v /var/run/docker.sock:/var/run/docker.sock`).
- cBot run / backtest containers run on **remote nodes** scheduled by `NodeScheduler` and
  driven over SSH by `SshContainerDispatcher`.
- The cTrader Console CLI's parameter passing path is not fully specified by upstream docs.
  We write parameter sets to `params.cbotset` (ini-style `key=value`) on the work dir; the
  exact way the CLI consumes them needs runtime verification.
- The MCP server is a separate process so it can be scaled or removed without touching the
  Web app.

## Things deliberately NOT done

- No optimization feature yet (cTrader Console doesn't support optimization).
- No email/SMTP integration — owner/admin reset passwords manually, user is forced to change
  on next login via `MustChangePassword`.
- Strong-typed EF value converters for all IDs — kept as `Guid` for now.
- Per-user quotas — out of scope for v1.

## How to extend

When adding a feature, prefer this order:

1. Add/extend an entity in `Core/Entities.cs` (inherit `AuditedEntity`).
2. Add a strong-typed ID for it in `Core/StrongIds.cs`.
3. Add constants (route, policy, etc.) to `Core/Constants/AppConstants.cs`.
4. Add a Minimal API endpoint group in `Web/Endpoints/`.
5. Add the corresponding Blazor page in `Web/Components/Pages/`.
6. If background work is involved, add a `BackgroundService` and wire via DI.
7. Regenerate the EF migration.
