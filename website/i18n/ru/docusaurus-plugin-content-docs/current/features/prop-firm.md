# Симуляция испытания проприетарной компании

Розничные пропь-компании (FTMO-стиль) продают **счета оценки**: трейдер должен достичь цели прибыли, оставаясь в пределах лимитов рисков (макс дневные потери, макс всего/трейлинг прорисовка, согласованность, временные лимиты) перед финансированием. cMind позволяет пользователю создать **пользовательский вызов любой формы промышленности**, привязать к `TradingAccount`, **запустить как операцию копирования торговли** — начало/остановка, хостируется на узле, отслеживается **в реальном времени через cTrader Open API**. Агрегат оценивает каждое правило детерминированно; при проход или нарушении, заканчивает вызов, отмечает, оповещает пользователя.

## Предметная область (ограниченный контекст: PropFirm)

`PropFirmChallenge` = корневой агрегат (модуль `Core.PropFirm`), ссылается на его `TradingAccount` по строгому id только (нет кросс-агрегата FK). Владеет оценкой правила, машиной фаза/состояние, арендой узла.

### Объекты значений и набор правил

- **`Money`** (не-отрицательные), **`MoneyAmount`** (подписанные), **`Percent`** (0–100], **`TradingDayRequirement`** (0–365).
- **`EquitySnapshot`** `(equity, balance)` — чтение подано на агрегат.
- **`ActivitySnapshot`** `(openPositions, openedInNewsWindow, holdingOverWeekend)` — не-капитальные факты.
- **`DailyLossLimit`** `(percent, basis)` — basis `Equity` (внутридневной, включает плавающий P&L) или `Balance` (реализовано только).
- **`DrawdownLimit`** — `Static` (от начального баланса), `TrailingPercent` (от пика капитала), или `TrailingThresholdDollar` (капиталы капитала пика на фиксированную сумму в долларах, затем **блокирует при начальном балансе** один раз капитал достигает порога — стиль фьючерсов).
- **`ConsistencyRule`** `(maxSingleDayShareOfProfit)` — блокирует проход пока один день доминирует общую прибыль.
- **`ChallengeRules`** носит выше плюс `MaxCalendarDays`, `MaxInactivityDays`, `MaxOpenPositions`, `AllowWeekendHolding`, `AllowNewsTrading`, `Kind`, `SingleStep`. Математика правила живет на VOs (`DrawdownLimit.IsBreached`, `DailyLossLimit.IsBreached`, `ConsistencyRule.IsSatisfied`); агрегат координирует.

### Типы вызовов и шаблоны

`ChallengeTemplates.For(kind)` построить действительный предустановок для `OnePhase`, `TwoPhase`, `ThreePhase`, `InstantFunding` или `Custom` (полный контроль). Пользовательский интерфейс предварительно заполняет шаблон; пользователь может отрегулировать любое поле.

### Фазы и статус

- **Фазы:** `Evaluation → Verification → Funded` (один-шаг пропускает Verification).
- **Статус:** `Active`, `Passed`, `Failed`, плюс жизненный цикл `Stopped` (отслеживание приостановлено) — `Create` начинает вызов `Active`; `Stop()`/`Resume()` переключение `Active↔Stopped`.
- **`BreachReason`:** `DailyLoss`, `MaxDrawdown`, `Consistency`, `TimeLimit`, `Inactivity`, `WeekendHolding`, `NewsTrading`, `MaxExposure`.

### Оценка правила

- **`RecordEquity(EquitySnapshot, now)`** — рулон торговый день на границах дня (захватывает прибыль предыдущего дня для правила согласованности), обновляет пик/дневные пики, затем **отказывает при первом нарушении** (дневные потери → прорисовка → временный лимит → неактивность, по порядку) или продвигает фазу когда цель прибыли, минимальный торговый день, требования согласованности все были встречены. Вне-порядок снимки и записи на терминальный вызов выбрасывают `DomainException`.
- **`RecordActivity(ActivitySnapshot, now)`** — оценивает правила поведения (макс открытые позиции, владение выходными, торговля новостями), штампует активность для правила неактивности.
- Мягкое **`PropFirmDrawdownWarning`** срабатывает один раз когда использование капитала пересекает сконфигурированный порог.

События предметной области: `PropFirmChallengeStarted`, `PropFirmChallengeStopped`, `PropFirmPhasePassed`, `PropFirmChallengePassed`, `PropFirmChallengeBreached`, `PropFirmDrawdownWarning`.
