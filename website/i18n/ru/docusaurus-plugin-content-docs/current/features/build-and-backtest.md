# Сборка и бэктестирование cBots

Сборка, запуск, бэктестирование cBots cTrader (C# **и** Python, оба .NET) из встроенной IDE Monaco, запуск на официальном образе `ghcr.io/spotware/ctrader-console`.

## Сборка

- Страница **Builder** хоста Editor Monaco; `CBotBuilder` компилирует проект с `dotnet build` **в одноразовый контейнер** (`AppOptions.BuildImage`, рабочий каталог привязан в `/work`), поэтому ненадёжные цели MSBuild пользователя не достигают хоста. Восстановление NuGet кэшируется во время сборок через общий том. Веб-хост нуждается в доступе сокета Docker.
- Стартовые шаблоны C# + Python живут в `src/Nodes/Builder/Templates/`.

## Запуск и бэктестирование

- **Экземпляры** = иерархия состояния TPH (`Run`/`Backtest` × `Pending`/`Scheduled`/`Starting`/`Running`/`Stopping`/`Stopped`/`Failed`). Переход заменяет сущность (изменение id), идентификатор контейнера переносится.
- `NodeScheduler` выбирает наименее нагруженный подходящий узел; `ContainerDispatcherFactory` маршрутизирует на удаленный HTTP-агент узла или локальный диспетчер Docker.
- Опросчики завершения согласовывают вышедшие контейнеры (контейнеры бэктеста само-выходят через `--exit-on-stop`); отчет присутствует → завершен (хранить `ReportJson`), отсутствует → неудачно.
- Логи контейнера в реальном времени передаются в браузер через SignalR; кривые капитала бэктеста разбираются из отчета и отображаются.

## Примечания к cTrader Console CLI

Бэктесты нуждаются в `--data-mode` (по умолчанию `m1`), даты как `dd/MM/yyyy HH:mm` и JSON позиционный аргумент `params.cbotset`; `run` отклоняет `--data-dir` (только бэктест). Смотрите `ContainerCommandHelpers`.

## Узлы и масштабирование

Емкость выполнения масштабируется путём добавления агентов узлов (самостоятельная регистрация + сердцебиение). Смотрите [обнаружение узлов](../operations/node-discovery.md) и [масштабирование](../deployment/scaling.md).
