# Сборка и тестирование cBots

Сборка, запуск и тестирование cBots cTrader (C# **и** Python, оба .NET) из встроенного браузерного редактора Monaco, запуск на официальном образе `ghcr.io/spotware/ctrader-console`.

## Сборка

- **Builder** — страница с редактором Monaco; `CBotBuilder` компилирует проект с помощью `dotnet build` **в одноразовом контейнере** (`AppOptions.BuildImage`, рабочая директория подключается как bind-mount в `/work`), чтобы недоверенные цели MSBuild не имели доступа к хосту. Восстановление NuGet кэшируется между сборками через общий volume. Веб-хосту требуется доступ к сокету Docker.
- Стартовые шаблоны C# и Python находятся в `src/Nodes/Builder/Templates/`.

## Запуск и тестирование

- **Instances** (экземпляры) = иерархия состояний TPH (`Run`/`Backtest` × `Pending`/`Scheduled`/`Starting`/`Running`/`Stopping`/`Stopped`/`Failed`). Переход заменяет сущность (изменяется id), id контейнера сохраняется.
- `NodeScheduler` выбирает наименее нагруженный подходящий узел; `ContainerDispatcherFactory` маршрутизирует на удаленный агент узла по HTTP или на локальный диспетчер Docker.
- Поллеры завершения согласовывают завершенные контейнеры (контейнеры тестирования самостоятельно завершаются через `--exit-on-stop`); отчет существует → завершено (сохранить `ReportJson`), отсутствует → ошибка.
- Живые логи контейнера передаются в браузер через SignalR; кривые капитала тестирования анализируются из отчета и отображаются на графике.

## Данные рыночного тестирования кэшируются на счет

Консоль cTrader загружает исторические тиковые/барные данные в свою директорию `--data-dir`. Эта директория — это **стабильный, постоянный кэш, ключом которого является торговый счет** (номер счета) — подключается как bind-mount с диска узла по его собственному пути контейнера (`/mnt/data`), **отдельный, не вложенный mount** от рабочей директории экземпляра. Поэтому каждое тестирование на том же счете **повторно использует** уже загруженные данные вместо их повторной загрузки при каждом запуске. (Ранее директория данных находилась в рабочей директории экземпляра, чей id меняется с каждым запуском, что заставляло загружать данные заново при каждом тестировании.) Эфемерная рабочая директория экземпляра по-прежнему содержит алгоритм, параметры, пароль и отчет; общий кэш данных учитывается в использовании данных тестирования узла и очищается действием очистки узла.

## Параметры тестирования

Диалог **Backtest** предоставляет настройки тестирования консоли cTrader, поддающиеся настройке пользователем, так чтобы вам никогда не нужно было трогать командную строку:

- **Symbol / Timeframe** (символ / интервал времени) — интервал времени — это **раскрывающееся меню каждого периода cTrader** (`t1`…`t1000`, `m1`…`m45`, `h1`…`h12`, `D1`/`D2`/`D3`, `W1`, `Month1` и периоды Renko/Range/Heikin), в каноническом регистре консоли, чтобы вы всегда выбрали допустимый `--period`.
- **From / To** (с / по) — окно тестирования (`--start` / `--end`).
- **Data mode** (режим данных) — один из трех режимов cTrader (`--data-mode`): **Tick data** (тиковые данные) (`tick`, точные), **m1 bars** (бары m1) (`m1`, быстро) или **Open prices only** (только цены открытия) (`open`, самое быстрое).
- **Starting balance** (начальный баланс) — по умолчанию `10000` (`--balance`). **Нулевой баланс не размещает сделки и заставляет cTrader выдать пустой отчет, на котором он затем падает** ("Message expected"), поэтому всегда отправляется ненулевой баланс.
- **Commission** (комиссия) и **Spread** (спред) — `--commission` / `--spread` (спред в пипсах).

Директория данных (`--data-file` / `--data-dir`) управляется самим приложением (кэш на счет, см. выше), не экспонируется в диалоге.

## Страница деталей экземпляра

Открытие экземпляра (`/instance/{id}`) показывает его живой статус, логи и — для тестирования — кривую капитала. **Название вкладки браузера** отражает конкретный экземпляр (**имя cBot · вид · символ**, например `TrendBot · Backtest · EURUSD`), так что вкладка живого запуска и вкладка тестирования различимы с первого взгляда. Запуск и тестирование одного cBot отслеживаются как различные **родословные** (стабильный id родословной, переносимый через переходы состояния), поэтому страница следует ровно одному экземпляру и никогда не смешивает данные запуска с данными тестирования.

## Управление жизненным циклом экземпляра

Каждая строка экземпляра (и его страница деталей) имеет управление, соответствующее состоянию. **Активный** экземпляр показывает **Stop** (остановка); **терминальный** (Stopped / Completed / Failed) показывает **Start (▶)** для его перезапуска с тем же cBot, счетом, символом, интервалом времени, набором параметров и образом (запуск перезапускается как запуск, тестирование как тестирование). Щелчок по Stop показывает уведомление "Stopping…" и отключает значок до его разрешения, и новый запуск сразу появляется в списке — без перезагрузки страницы.

Логи консоли **сохраняются при завершении экземпляра** — для запуска (на Stop) и для **тестирования** (при завершении) — поэтому логи последнего запуска остаются доступны для просмотра на странице деталей и, через панель инструментов логирования, **скопированы в буфер обмена** (значок копирования логов) или **загружены** (значок загрузки логов) даже после того, как контейнер исчез. Оба действуют на полный журнал консоли экземпляра, а не только на видимый хвост.

Загруженный `.algo` никогда не был создан здесь, поэтому его столбец **Last Build** на странице cBots остается пустым (он показывает время построения только для cBots, которые вы создаете в браузере).

## Редактирование и перезапуск остановленного экземпляра

**Остановленный** экземпляр (запуск или тестирование) имеет элемент управления **Edit** (редактирование) — значок в его строке в списке **и** рядом с Start/Stop на его странице деталей — который открывает диалог **предзаполненный** его текущей конфигурацией. Вы можете изменить **торговый счет, символ, интервал времени, набор параметров и тег образа** (и, для тестирования, **окно и все параметры тестирования** выше), затем **Save & start** перезапускает его с новыми параметрами (заменяя остановленный экземпляр). Элемент управления **отключен, пока экземпляр активен** — только остановленный экземпляр может быть отредактирован.

## Запуск из редактора кода

Щелчок **Run** в редакторе кода открывает диалог вместо слепого, жестко закодированного запуска:

- **Trading account** (торговый счет) (требуется) — счет cTrader, к которому подключается cBot.
- **Parameter set** (набор параметров) (опционально) — выберите существующий набор или оставьте его пустым для запуска с **значениями параметров cBot по умолчанию**. Кнопка **+** рядом с селектором создает новый набор параметров встроенным образом (см. ниже) и выбирает его.
- **Symbol / Timeframe** (символ / интервал времени) по умолчанию установлены на `EURUSD` / `h1` и могут быть изменены; **Cancel** (отмена) или **Run** (запуск).

На **Run** редактор сохраняет + создает текущий источник, запускает экземпляр на выбранном счете с выбранными параметрами, затем отслеживает живые логи контейнера. (Поток логов передает куку аутентификации вошедшего пользователя на хаб SignalR `/hubs/logs`, поэтому он подключается вместо отказа с `Invalid negotiation response received`.)

## Наборы параметров

**Набор параметров** — это именованный, многоразовый набор переопределений параметров cBot, хранящихся как плоский JSON объект, сопоставляющий каждое имя параметра со скалярным значением, например `{"Period": 14, "Label": "trend"}`. Во время запуска/тестирования он превращается в файл cTrader `params.cbotset` (`{ "Parameters": { … } }`). Вы можете создать/отредактировать набор как необработанный JSON из диалога **Parameter sets** cBot или встроенным образом из диалога запуска.

Каждый набор параметров **принадлежит cBot**: диалог New Parameter Set перечисляет все ваши cBots и вы **должны выбрать один** — создание блокируется до тех пор, пока cBot не будет выбран. **Имя набора уникально для каждого cBot**: создание или переименование набора в имя, которое уже использует другой набор того же cBot, отклоняется (четкая ошибка в диалоге, `409 Conflict` в API). То же имя может быть повторно использовано на **другом** cBot.

JSON **проверяется** при сохранении: это должен быть единственный плоский объект, значения которого все скалярные (строка / число / логическое значение). Корень не-объект, массив, вложенный объект, значение `null` или неправильный JSON отклоняются (четкая ошибка в диалоге, `400 Bad Request` в API). Пустой объект `{}` разрешен и означает "без переопределений".

## Примечания к CLI консоли cTrader

Тестирование требует `--data-mode` (по умолчанию `m1`), даты как `dd/MM/yyyy HH:mm` и JSON-аргумент `params.cbotset` позиционный; `run` отклоняет `--data-dir` (только тестирование). См. `ContainerCommandHelpers`.

## Узлы и масштабирование

Пропускная способность выполнения масштабируется путем добавления агентов узлов (самостоятельная регистрация + периодическая передача). См. [node discovery](../operations/node-discovery.md) и [scaling](../deployment/scaling.md).

## Требуется торговый счет

Запуск или тестирование cBot требует торговый счет cTrader для подключения. До тех пор, пока вы не добавите один в **Trading accounts**, кнопки **Run New cBot** / **Backtest New cBot** отключены (с подсказкой) и страница показывает приглашение со ссылкой на настройку счета — вы больше не получите ошибку `stream connect failed` от бота без счета.
