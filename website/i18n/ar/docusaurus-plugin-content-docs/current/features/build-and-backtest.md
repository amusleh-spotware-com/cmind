# بناء واختبار cBots

بناء وتشغيل واختبار cBots من cTrader (C# **و** Python، كليهما .NET) من محرر Monaco
المدمج في المتصفح، التشغيل على صورة `ghcr.io/spotware/ctrader-console` الرسمية.

## البناء

- صفحة **Builder** تستضيف محرر Monaco؛ `CBotBuilder` يترجم المشروع باستخدام
  `dotnet build` **في حاوية مؤقتة** (`AppOptions.BuildImage`، دليل العمل يُربط في
  `/work`)، لذا لا يمكن لأهداف MSBuild غير الموثوقة الوصول إلى المضيف. يتم تخزين استعادة NuGet مؤقتاً
  عبر الأشغال عبر مجلد مشترك. يحتاج مضيف الويب إلى وصول مأخذ Docker.
- نماذج البداية لـ C# و Python موجودة في `src/Nodes/Builder/Templates/`.

## التشغيل والاختبار

- **Instances** = TPH تسلسل هرمي للحالة (`Run`/`Backtest` × `Pending`/`Scheduled`/`Starting`/
  `Running`/`Stopping`/`Stopped`/`Failed`). الانتقال يحل محل الكيان (تغيير المعرف)،
  معرف الحاوية يُنقل معه.
- `NodeScheduler` يختار العقدة الأقل تحميلاً المؤهلة؛ `ContainerDispatcherFactory` يوجه إلى
  وكيل HTTP العقدة البعيدة أو موزع Docker المحلي.
- تصالحات المكتمل تصالح الحاويات المغلقة (حاويات الاختبار تغلق نفسها عبر
  `--exit-on-stop`)؛ التقرير موجود → مكتمل (تخزين `ReportJson`)، مفقود → فشل.
- سجلات الحاوية المباشرة تتدفق إلى المتصفح عبر SignalR؛ منحنيات الأسهم للاختبار يتم تحليلها من
  التقرير + رسم بياني.

## بيانات السوق للاختبار يتم تخزينها مؤقتاً لكل حساب

يقوم cTrader Console بتنزيل بيانات tick/bar التاريخية إلى `--data-dir`. هذا الدليل هو
**مخزن مؤقت مستقر وثابت مفتاح على حساب التداول** (رقم الحساب) — مرتبط من قرص العقدة في مسار الحاوية الخاص بها (`/mnt/data`)، **ربط منفصل غير متداخل** من
دليل العمل لكل نسخة. لذا كل اختبار على نفس الحساب **إعادة استخدام** البيانات المنزلة مسبقاً
بدلاً من إعادة تنزيلها كل مسار. (في السابق كان دليل
البيانات يعيش تحت دليل العمل لكل نسخة، الذي يتغير معرفه كل مسار، مما أجبر على تنزيل طازج
كل اختبار.) دليل العمل العابر لكل نسخة يزال يحمل الخوارزمية والمعاملات وكلمة المرور
والتقرير؛ مخزن البيانات المشترك يتم عده في استخدام بيانات الاختبار للعقدة ويتم حذفه بواسطة
إجراء التنظيف على مستوى العقدة.

## إعدادات الاختبار

حوار **Backtest** يعرض إعدادات اختبار cTrader Console القابلة للتعديل من قبل المستخدم، لذا لا تحتاج أبداً إلى
لمس سطر الأوامر:

- **Symbol / Timeframe** — الإطار الزمني هو **قائمة منسدلة لكل فترة cTrader** (`t1`…`t1000`،
  `m1`…`m45`، `h1`…`h12`، `D1`/`D2`/`D3`، `W1`، `Month1`، وفترات Renko/Range/Heikin)، في
  الحالة الكنسية للوحدة الطرفية، لذا تختار دائماً `--period` صالح.
- **From / To** — نافذة الاختبار (`--start` / `--end`).
- **Data mode** — أحد الأنماط الثلاثة لـ cTrader (`--data-mode`): **بيانات Tick** (`tick`، دقيق)،
  **أعمدة m1** (`m1`، سريع)، أو **أسعار الفتح فقط** (`open`، الأسرع).
- **الرصيد الأولي** — يفترض `10000` (`--balance`). **رصيد 0 لا يضع أي صفقات ويجعل
  cTrader تصدر تقرير فارغ ثم تتعطل عليه** ("Message expected")، لذا يتم إرسال رصيد غير صفري
  دائماً.
- **Commission** و **Spread** — `--commission` / `--spread` (الفارق في نقاط).

دليل البيانات (`--data-file` / `--data-dir`) يُدار من قبل التطبيق نفسه (مخزن مؤقت لكل حساب، انظر
أعلاه)، وليس معروضاً في الحوار.

## صفحة تفاصيل النسخة

فتح نسخة (`/instance/{id}`) يعرض حالتها المباشرة والسجلات و — لاختبار — منحنى الأسهم.
**عنوان تبويب المتصفح** يعكس نسخة معينة (**اسم cBot · النوع · الرمز**، على سبيل المثال
`TrendBot · Backtest · EURUSD`) لذا تبويب التشغيل المباشر وتبويب الاختبار يمكن تمييزهما في لمحة.
تشغيل واختبار نفس cBot يتم تتبعهما كـ **نسب** مختلفة (معرف نسب مستقر يُنقل
عبر انتقالات الحالة)، لذا الصفحة تتبع نسخة واحدة بالضبط ولا تخلط بيانات التشغيل مع
اختبار.

## عناصر تحكم دورة حياة النسخة

كل صف نسخة (وصفحة تفاصيلها) له عناصر تحكم صحيحة للحالة. نسخة **نشطة** تعرض
**Stop**؛ نسخة **نهائية** (Stopped / Completed / Failed) تعرض **Start (▶)** لإعادة تشغيلها برفقة
نفس cBot والحساب والرمز والإطار الزمني ومجموعة المعاملات والصورة (تشغيل يعاد تشغيله كتشغيل، اختبار كاختبار). النقر على Stop يعرض إشعار "Stopping…" ويعطل الرمز
حتى يحل، ويظهر تشغيل جديد تم إنشاؤه في القائمة فوراً — لا حاجة لإعادة تحميل الصفحة.

سجلات الوحدة الطرفية يتم **حفظها عند انتهاء نسخة** — لتشغيل (على Stop) و
**اختبار** (عند الانتهاء) على حد سواء — لذا سجلات آخر تشغيل تبقى قابلة للعرض على صفحة التفاصيل و،
عبر شريط أدوات السجل، **نسخ إلى الحافظة** (أيقونة نسخ السجلات) أو **تحميل** (أيقونة تحميل السجلات)
حتى بعد اختفاء الحاوية. كلاهما يعمل على سجل الوحدة الطرفية الكامل للنسخة، وليس فقط
الذيل على الشاشة.

`.algo` **تم تحميله** لم يتم بناؤه هنا، لذا عمود **آخر بناء** على صفحة cBots **يُترك فارغاً** (يعرض وقت بناء فقط لـ cBots التي تبنيها في المتصفح).

## تحرير وإعادة تشغيل نسخة متوقفة

نسخة **متوقفة** (تشغيل أو اختبار) لها عنصر تحكم **Edit** — أيقونة على صفها في القائمة **و**
بجانب Start/Stop على صفحة تفاصيلها — التي تفتح حواراً **مملوءاً مسبقاً** بإعدادها الحالية.
يمكنك تغيير **حساب التداول والرمز والإطار الزمني ومجموعة المعاملات وعلامة الصورة** (و، لاختبار،
**النافذة وكل إعدادات الاختبار** أعلاه)، ثم **Save & start** يعيد تشغيلها بالإعدادات
الجديدة (استبدال النسخة المتوقفة). العنصر **معطل بينما النسخة نشطة** —
فقط نسخة متوقفة يمكن تحريرها.

## التشغيل من محرر الأكواد

النقر على **Run** في محرر الأكواد يفتح حواراً بدلاً من إطلاق تشغيل عمياء مبرمجة:

- **Trading account** (مطلوب) — حساب cTrader الذي يتصل به cBot.
- **Parameter set** (اختياري) — اختر مجموعة موجودة، أو اتركها فارغة للتشغيل برفقة
  **قيم المعاملات الافتراضية** للـ cBot. زر **+** بجانب المحدد ينشئ مجموعة معاملات جديدة
  مضمناً (انظر أدناه) ويختارها.
- **Symbol / Timeframe** تافتراضان إلى `EURUSD` / `h1` ويمكن تغييرهما؛ **Cancel** أو **Run**.

على **Run** المحرر يحفظ + يبني المصدر الحالي، يبدأ النسخة على الحساب المختار
بالمعاملات المختارة، ثم يتابع سجلات الحاوية المباشرة. (تدفق السجل يوجه ملف تعريف الارتباط المصادق عليه للمستخدم إلى مركز SignalR `/hubs/logs`، لذا يتصل بدلاً من الفشل برفقة
`Invalid negotiation response received`.)

## مجموعات المعاملات

**مجموعة معاملات** هي مجموعة مسماة وقابلة لإعادة الاستخدام من تجاوزات معاملات cBot مُخزنة كـ JSON
كائن مسطح يربط كل اسم معامل إلى قيمة قياسية، على سبيل المثال `{"Period": 14, "Label": "trend"}`. في
وقت التشغيل/الاختبار يتم تحويله إلى ملف `params.cbotset` من cTrader
(`{ "Parameters": { … } }`). يمكنك إنشاء/تحرير مجموعة كـ JSON خام من حوار **مجموعات المعاملات** الخاص بـ cBot أو مضمناً من حوار التشغيل.

كل مجموعة معاملات **تنتمي إلى cBot**: حوار New Parameter Set يسرد كل cBots الخاصة بك و يجب
**اختيار واحد** — الإنشاء يُحجب حتى يتم اختيار cBot. **اسم** مجموعة **فريد لكل cBot**:
إنشاء أو إعادة تسمية مجموعة بـ اسم مجموعة أخرى من نفس cBot تستخدمه بالفعل يُرفض (خطأ واضح في الحوار، `409 Conflict` على API). الاسم نفسه قد يُعاد استخدامه على **cBot مختلف**.

JSON **يُتحقق من** عند الحفظ: يجب أن يكون كائن مسطح واحد قيمه كلها قياسية
(سلسلة نصية / رقم / منطقي). جذر غير كائن أو مصفوفة أو كائن متداخل أو قيمة `null` أو
JSON مشوه يُرفض (خطأ واضح في الحوار، `400 Bad Request` على API). كائن فارغ `{}`
مسموح ويعني "لا توجد تجاوزات".

## ملاحظات CLI لـ cTrader Console

الاختبارات تحتاج `--data-mode` (افتراضي `m1`)، تواريخ `dd/MM/yyyy HH:mm`، و
معامل JSON موضعي لـ `params.cbotset`؛ `run` ترفض `--data-dir` (اختبار فقط). انظر
`ContainerCommandHelpers`.

## العقد والتوسع

القدرة على التنفيذ توسع بإضافة وكلاء عقد (تسجيل ذاتي + نبض قلب). انظر
[كشف العقدة](../operations/node-discovery.md) و [التوسع](../deployment/scaling.md).

## حساب تداول مطلوب

تشغيل أو اختبار cBot يحتاج إلى حساب تداول من cTrader للاتصال به. حتى تضيف واحد تحت
**Trading accounts**، زري **Run New cBot** / **Backtest New cBot** معطلان (برفقة
تلميح) والصفحة تعرض موجه يربط إلى إعداد الحساب — لا تضرب بخطأ `stream connect failed` الخام
من bot بدون حساب.
