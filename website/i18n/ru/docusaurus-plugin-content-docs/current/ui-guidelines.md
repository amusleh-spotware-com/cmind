# Рекомендации по дизайну UI — ОБЯЗАТЕЛЬНО

Обязательные требования к **каждому** новому или изменённому элементу UI в этом приложении (Blazor-страницы, диалоги, компоненты).
Это источник истины, на который ссылается `CLAUDE.md`. Если правило блокирует вас, остановитесь и спросите — не отгружайте
UI, который его нарушает. Основано на `plans/ui-overhaul.md`.

## 1. Mobile-first, всегда

- **Проектируйте сначала для 360–430px телефона**, затем улучшайте для более крупных экранов через `min-width` media queries /
  свойства breakpoints MudBlazor. Никогда не проектируйте desktop-first с `max-width` переопределениями.
- **Нет горизонтальной прокрутки ни на какой ширине 320–1920px.** Если контент шире viewport — это баг.
- Touch-цели ≥ **44px** (`var(--app-touch-target)`). Текстовые поля ввода ≥ 16px шрифт (предотвращает iOS zoom-on-focus).
- Уважайте вырезы (notches): используйте `env(safe-area-inset-*)`; viewport уже установлен `viewport-fit=cover`.
- Уважайте `prefers-reduced-motion` — никакая важная информация не должна передаваться только анимацией.

## 2. Дизайн-токены — никаких hard-coded значений

- Все цвета/радиусы/отступы берутся из **дизайн-токенов**: тема MudBlazor (`Web/Components/Theme.cs`) +
  CSS custom properties от `Web/Branding/BrandingCss.cs` (`var(--app-primary)`,
  `--app-surface`, `--app-border`, `--app-text*`, `--app-radius`, …).
- **Никогда не пишите hard-coded hex-цвет, радиус или брендовую строку в компоненте или CSS-правиле.** Читайте токен.
  Токены приходят от white-label `BrandingOptions`, поэтому палитра реселлера должна бесплатно достигать вашего UI.
- Новое значение, влияющее на бренд → добавьте токен + поле брендинга; не вставляйте inline.

## 3. Адаптивная вёрстка и данные

- **Таблицы сворачиваются в карточки на телефонах.** Каждая `MudTable` имеет `Breakpoint="Breakpoint.Sm"` и каждая
  `MudTd` имеет `DataLabel`. Никакихraw широких таблиц на мобильных. (Шаблон: `Components/Pages/Nodes.razor`.)
- Сетки: `MudItem xs="12" sm="6" md="4"` — на телефоне полная ширина, несколько колонок на больших экранах.
- Формы в одну колонку на мобильных; крупные touch-цели; `inputmode`/`autocomplete` на полях ввода; numeric/decimal
  inputmode для денег/процентов.
- Предусматривайте состояния **загрузки, пустоты и ошибки** для каждого списка/детали — размер под мобильный.
- Мобильная **нижняя навигация** (`Components/Layout/BottomNav.razor`) — это основная навигация на телефоне;
  сгруппированный drawer — полное меню. Добавляйте туда высокопосещаемые пункты; не больше 5 элементов.

## 4. Диалоги (создание/редактирование)

- Все действия добавления/создания/редактирования используют **MudBlazor диалог** (`IDialogService.ShowAsync<TDialog>`), никогда
  inline-форму на странице. Диалоги живут в `Web/Components/Dialogs/`, предоставляют `[Parameter]`ы, возвращают вложенный
  `public sealed record …Result(...)`. Действия строки (start/stop/delete) остаются inline как иконки-кнопки.
- На телефонах диалоги должны быть **full-screen / full-width** и учитывать клавиатуру.

## 5. Inline-справка — каждый контрол

- Каждый неочевидный параметр, select, switch или действие получает **`<HelpTip Text="…" />`**
  (`Components/HelpTip.razor`) — hover на desktop, **tap на мобильном**. Источник текста из `docs/` чтобы
  руководство синхронизировалось с поведением; обновляйте оба в одном коммите.

## 6. White-label

- Название продукта, логотип, описание, поддержка/компания, цвета, favicon — всё из `BrandingOptions`.
  Ссылайтесь на них (`IBrandingThemeProvider` / `IOptionsMonitor<AppOptions>`), никогда не пишите литерал "cMind"
  или брендовый цвет. PWA-манифест, иконки, theme-color и login hero — всё брендировано.

## 7. PWA

- Приложение устанавливаемое. Манифест endpoint (`/manifest.webmanifest`) брендирован, иконки на месте
  (192/512/maskable + apple-touch), service worker только app-shell (никогда не трогает Blazor
  circuit/`_framework`/hubs), и offline-страница работает. Новый статический route → сохраняйте `scope` манифеста.
- Blazor Server нужен живой SignalR circuit → **устанавливаемое + app-shell**, не полный offline. Не обещайте
  offline-интерактивность.

## 8. Доступность

- Labels на полях ввода, `aria-*` на кастомных контролах, видимый фокус, логичный порядок фокуса. Поскольку тема
  white-label, проверяйте **контраст** на активной теме, а не на фиксированной палитре.

## 9. E2E — никакой UI не отгружается без тестов (блокирующе)

Каждое пользовательское изменение отгружается с Playwright E2E в `tests/E2ETests`, запускается как реальный пользователь,
**на эмуляции мобильного устройства** плюс desktop:

- Новый route → добавьте его в `PageSmokeTests` **и** `MobileLayoutTests` (отображается, bottom nav, нет error UI).
- Преобразование таблицы/страницы → добавьте её route в мобильный **no-overflow** набор.
- Новый flow → реалистичное мобильное путешествие (create/edit/save round-trip) **и** unhappy path
  (некорректный ввод, пустой список, permission-denied per role).
- Новый help tip → assert он открывается на tap (`HelpTipTests` pattern).
- Используйте `AppFixture.NewAuthedMobilePageAsync` / `NewAnonymousMobilePageAsync` (эмуляция устройства).
- `dotnet test` зелёный перед "done". Эмулируемый WebKit ≠ мобильный Safari — gating на реальном устройстве это отдельный
  этап release.

## 10. Definition of done (UI)

- [ ] Mobile-first; нет горизонтального overflow 320–1920px; touch-цели ≥44px.
- [ ] Только дизайн-токены — ноль hard-coded цветов/радиусов/брендовых строк.
- [ ] Таблицы → карточки на телефоне (`DataLabel` + `Breakpoint.Sm`); состояния загрузки/пустоты/ошибки присутствуют.
- [ ] Create/edit через диалог; full-screen на мобильных.
- [ ] У каждого контрола есть `HelpTip` источник из docs.
- [ ] White-label + PWA уважаются.
- [ ] Mobile + desktop E2E добавлены (smoke, no-overflow, journey, unhappy path); `dotnet test` зелёный.
- [ ] Rider `get_file_problems` + `dotnet format analyzers` чистые на затронутых файлах.
