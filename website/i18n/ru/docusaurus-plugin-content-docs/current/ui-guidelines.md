# Руководство по дизайну пользовательского интерфейса — ОБЯЗАТЕЛЬНО

Связь для **каждой** новой или изменённой части пользовательского интерфейса в этом приложении (страницы Blazor, диалоги, компоненты). Это источник истины, упоминаемый `CLAUDE.md`. Если правило блокирует вас, остановитесь и спросите — не поставляйте пользовательский интерфейс, нарушающий это. Имеет корни в `plans/ui-overhaul.md`.

## 1. Мобильный приоритет, всегда

- **Создавайте для телефона 360–430px в первую очередь**, затем улучшайте с помощью `min-width` медиа-запросов / реквизитов контрольной точки MudBlazor. Никогда не первый рабочий стол с переопределениями `max-width`.
- **Без горизонтальной прокрутки при любой ширине 320–1920px.** Если содержимое шире видового окна, это ошибка.
- Цели сенсорной нажатия ≥ **44px** (`var(--app-touch-target)`). Текстовые входы ≥ 16px шрифт (останавливает фокус масштабирования iOS).
- Учитывайте вырезки: используйте `env(safe-area-inset-*)`; вьюпорт уже устанавливает `viewport-fit=cover`.
- Честь `prefers-reduced-motion` — никакая важная информация не передаётся только анимацией.

## 2. Токены дизайна — нет жёстко кодированных значений

- Все цвета/радиус/интервалы поступают из **токенов дизайна**: тема MudBlazor (`Web/Components/Theme.cs`) + пользовательские свойства CSS, излучаемые `Web/Branding/BrandingCss.cs` (`var(--app-primary)`, `--app-surface`, `--app-border`, `--app-text*`, `--app-radius`, …).
- **Никогда не жёстко кодируйте цвет hex, радиус или строку бренда в компоненте или правиле CSS.** Прочитайте токен. Токены текут из `BrandingOptions` белого ярлыка, поэтому палитра перепродавца должна достичь вашего пользовательского интерфейса бесплатно.
- Новое значение, влияющее на бренд → добавьте токен + поле брендирования; не встраивайте его.

## 3. Адаптивная компоновка и данные

- **Таблицы сворачиваются в карты на телефонах.** Каждый `MudTable` устанавливает `Breakpoint="Breakpoint.Sm"` и каждый `MudTd` имеет `DataLabel`. Нет сырой широкой таблицы на мобильном. (Шаблон: `Components/Pages/Nodes.razor`.)
- Сетки: `MudItem xs="12" sm="6" md="4"` — полная ширина на телефоне, мультиколонка вверх.
- Формы одна колонка на мобильном; большие цели касания; `inputmode`/`autocomplete` на входах; числовой/десятичный inputmode для денег/процентов.
- Предоставьте **загрузку, пустоту и ошибку** состояния на каждом списке/детали — рассчитанные на мобильный.
- Мобильная **нижняя навигация** (`Components/Layout/BottomNav.razor`) — основная навигация по телефону; подшивка группировки — полное меню. Добавляйте высокотрафик направления туда; держите ≤5 пункты.

## 4. Диалоги (создание/редактирование)

- Все действия добавления/создания/редактирования/новые используют **диалог MudBlazor** (`IDialogService.ShowAsync<TDialog>`), никогда не встроенная форма страницы. Диалоги живут в `Web/Components/Dialogs/`, раскрывают `[Parameter]`, возвращают вложенные `public sealed record …Result(...)`. Действия строк списков (начало/остановка/удаление) остаются встроенными как кнопки значков.
- На телефонах диалоги должны быть **полноэкранными / полной ширины** и с учетом клавиатуры.

## 5. Встроенная справка — каждый контрольный элемент

- Каждая неочевидная опция, выбор, переключатель или действие получает **`<HelpTip Text="…" />`** (`Components/HelpTip.razor`) — наведение на рабочий стол, **касание на мобильном**. Введите текст из `docs/` чтобы руководство оставалось в синхронизации с поведением; обновите оба в один commit.

## 6. Белый ярлык

- Имя продукта, логотип, описание, поддержка/компания, цвета, favicon все поступают из `BrandingOptions`. Ссылайтесь на них (`IBrandingThemeProvider` / `IOptionsMonitor<AppOptions>`), никогда не литеральные "cMind" или цвет бренда. Манифест PWA, значки, theme-color и герой входа все брендированы.
