# cBots のビルドとバックテスト

cTrader cBots（C# **および** Python、両方ともリリース）をブラウザ内 Monaco
IDE からビルド、実行、バックテストし、公式 `ghcr.io/spotware/ctrader-console` イメージで実行します。

## ビルド

- **Builder** ページは Monaco エディタをホストします。`CBotBuilder` は
  `dotnet build` で **一時的なコンテナ内で** プロジェクトをコンパイルします（`AppOptions.BuildImage`、作業ディレクトリは
  `/work` でバインドマウントされます）。これにより、信頼できないユーザーの MSBuild ターゲットがホストに到達しないようになります。NuGet 復元は共有ボリュームを介してビルド間でキャッシュされます。Web ホストは Docker ソケット アクセスが必要です。
- C# および Python スターター テンプレートは `src/Nodes/Builder/Templates/` にあります。

## 実行とバックテスト

- **Instances** = TPH 状態階層（`Run`/`Backtest` × `Pending`/`Scheduled`/`Starting`/
  `Running`/`Stopping`/`Stopped`/`Failed`）。遷移はエンティティを置き換えます（ID 変更）。
  コンテナ ID は引き継がれます。
- `NodeScheduler` は最も負荷の低い適格ノードを選択します。`ContainerDispatcherFactory` は
  リモート ノード HTTP エージェントまたはローカル Docker ディスパッチャにルーティングします。
- 完了ポーラーは終了したコンテナを調整します（バックテスト コンテナは
  `--exit-on-stop` を介して自動的に終了します）。レポートが存在 → 完了（`ReportJson` を保存）、欠落 → 失敗。
- ライブ コンテナ ログは SignalR を介してブラウザにストリーミングされます。バックテスト エクイティ曲線はレポートから解析され、グラフ化されます。

## バックテスト市場データはアカウントごとにキャッシュされます

cTrader Console は履歴ティック/バー データを `--data-dir` にダウンロードします。そのディレクトリは
**取引アカウント（アカウント番号）をキーにした安定した永続キャッシュ** です。ノードのディスクからノード独自のコンテナ パス（`/mnt/data`）にバインドマウントされ、
インスタンスごとの作業ディレクトリとは **別の、ネストされていないマウント** です。したがって、同じアカウント上のすべてのバックテストは
既にダウンロード済みのデータを **再利用** し、実行のたびに再ダウンロードされません。（以前、
データ ディレクトリはインスタンスごとの作業ディレクトリの下にありました。そのため、その ID は実行のたびに変更され、各バックテストで新しいダウンロードが強制されました。）一時的なインスタンスごとの作業ディレクトリは、アルゴリズム、パラメーター、パスワード、
レポートをまだ保持しています。共有データ キャッシュはノードのバックテスト データ使用量にカウントされ、ノード クリーン アクションでクリアされます。

## バックテスト設定

**Backtest** ダイアログは、cTrader Console バックテスト CLI が受け入れるすべての設定を公開するため、
コマンド ラインに触れる必要はありません：

- **From / To** — バックテスト ウィンドウ（`--start` / `--end`）。
- **Data mode** — `m1`（1分足）または `tick`（`--data-mode`）。
- **Starting balance** — デフォルトは `10000`（`--balance`）。**残高が 0 の場合、取引が行われず、cTrader が空のレポートを出力してクラッシュします**（「メッセージが必要」）。したがって、ゼロ以外の残高が常に送信されます。
- **Commission** および **Spread**（`--commission` / `--spread`、スプレッドはピップス単位）。
- **Advanced options** — cTrader がサポートする他のバックテスト オプション（例：`applyCommissionAutomatically=true`）の場合、1 行あたり 1 つの `name=value` フリーフォーム ボックス。各行は `--name value` CLI 引数になります。

## インスタンス詳細ページ

インスタンス（`/instance/{id}`）を開くと、そのライブ ステータス、ログ、およびバックテストの場合はエクイティ曲線が表示されます。**ブラウザ タブのタイトル** は特定のインスタンスを反映します（**cBot 名 · 種類 · シンボル**、例：
`TrendBot · Backtest · EURUSD`）。ため、ライブ実行タブとバックテスト タブは一目で区別できます。
同じ cBot の実行とバックテストは、個別の **系統** として追跡されます（状態遷移全体で実行される安定した系統 ID）。ページは正確に 1 つのインスタンスに従い、実行のデータをバックテストのデータと混ぜません。

## インスタンス ライフサイクル コントロール

各インスタンス行（およびその詳細ページ）には、状態に適したコントロールがあります。**アクティブな** インスタンスは
**Stop** を表示します。**ターミナル** インスタンス（Stopped / Completed / Failed）は **Start（▶）** を表示して、同じ cBot、アカウント、シンボル、タイムフレーム、パラメーター セット、イメージで再起動します（実行は実行として再起動し、バックテストはバックテストとして再起動します）。Stop をクリックすると「Stopping…」通知が表示されアイコンが無効になるまで待機され、新しく作成された実行がリストに直ちに表示されます。ページを再読み込みする必要はありません。

コンソール ログは **インスタンスが終了したときに永続化されます** — 実行（停止時）とバックテスト（完了時）の両方で — そのため、最後の実行のログは詳細ページで表示したままであり、ログ ツールバーを使用して、**クリップボードにコピー** され（ログをコピー アイコン）または **ダウンロード** されます（ログをダウンロード アイコン）。コンテナが消えた後でも。両方ともインスタンスの完全なコンソール ログに作用し、画面上のテールだけではありません。

アップロード された `.algo` はここで構築されなかったため、cBots ページの **Last Build** 列は空白のままです（ブラウザでビルドした cBots のビルド時間のみを表示します）。

## 停止したインスタンスを編集して再実行する

**停止した** インスタンス（実行またはバックテスト）には **Edit** コントロールがあります — リスト内の行の アイコン **および**
詳細ページの Start/Stop の隣 — 現在の構成で **事前入力された** ダイアログを開きます。
**取引アカウント、シンボル、タイムフレーム、パラメーター セット、イメージ タグ** （およびバックテストの場合、
**ウィンドウとすべてのバックテスト設定** ）を変更し、**Save & start** は新しい設定で再起動します（停止したインスタンスを置き換えます）。このコントロールは **インスタンスがアクティブな間は無効です** — 停止したインスタンスのみを編集できます。

## コード エディタから実行

コード エディタの **Run** をクリックすると、ブラインド、ハードコード実行を実行する代わりにダイアログが開きます：

- **Trading account**（必須）— cBot が接続する cTrader アカウント。
- **Parameter set**（オプション）— 既存のセットを選択するか、空のままにして cBot の
  **デフォルト パラメーター値** で実行します。セレクタの隣の **+** ボタンは、新しいパラメーター セットをインラインで作成し（下記参照）、それを選択します。
- **Symbol / Timeframe** は `EURUSD` / `h1` にデフォルト設定されており、変更できます。**Cancel** または **Run**。

**Run** ではエディタが現在のソースを保存してビルドし、選択したアカウント上の選択したパラメーターでインスタンスを開始し、ライブ コンテナ ログをテールします。（ログ ストリームはサインインしたユーザーの認証クッキーを `/hubs/logs` SignalR ハブに転送するため、`Invalid negotiation response received` で失敗する代わりに接続されます。）

## パラメーター セット

**parameter set** は、名前付きで再利用可能な cBot パラメーター オーバーライドのセットであり、フラット JSON
オブジェクトとして保存されており、各パラメーター名をスカラー値にマップしています（例：`{"Period": 14, "Label": "trend"}`）。実行/バックテスト時には cTrader `params.cbotset` ファイルに変換されます
（`{ "Parameters": { … } }`）。cBot の **Parameter
sets** ダイアログから生のJSONとして、または Run ダイアログからインラインでセットを作成/編集できます。

すべてのパラメーター セットは **cBot に属しています** — New Parameter Set ダイアログはすべての cBot をリストし、
**1 つを選択する必要があります** — cBot が選択されるまで作成がブロックされます。セットの **名前は cBot ごとに一意です** —
セットを別のセットが既に使用している名前に作成または名前変更すると、拒否されます（ダイアログで明確なエラー、API で `409 Conflict`）。同じ名前は **別の** cBot で再利用できます。

JSON は **保存時に検証されます** — 単一のフラット オブジェクトである必要があり、その値はすべてスカラーです
（文字列 / 数値 / ブール値）。非オブジェクト ルート、配列、ネストされたオブジェクト、`null` 値、または不正な形式の
JSON は拒否されます（ダイアログで明確なエラー、API で `400 Bad Request`）。空のオブジェクト `{}`
は許可され、「オーバーライドなし」を意味します。

## cTrader Console CLI に関する注記

バックテストには `--data-mode`（デフォルト `m1`）、日付が `dd/MM/yyyy HH:mm` の形式、および
`params.cbotset` JSON 位置指定引数が必要です。`run` は `--data-dir` を拒否します（バックテストのみ）。`ContainerCommandHelpers` を参照してください。

## ノードとスケーリング

実行容量はノード エージェントを追加することによってスケールします（自動登録 + ハートビート）。
[ノード検出](../operations/node-discovery.md) と [スケーリング](../deployment/scaling.md) を参照してください。

## 取引アカウントが必須

cBot を実行またはバックテストするには、接続するための cTrader 取引アカウントが必要です。**Trading accounts** の下に 1 つを追加するまで、
**Run New cBot** / **Backtest New cBot** ボタンは無効になります（ツールチップ付き）。ページはアカウント セットアップにリンクするプロンプトを表示します。アカウントのない bot から生の `stream connect failed` エラーが発生することはなくなりました。
